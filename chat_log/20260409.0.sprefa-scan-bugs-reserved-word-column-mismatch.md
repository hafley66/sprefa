# Session: sprefa-scan-bugs-reserved-word-column-mismatch

## Goal
Get sprefa scanning its own repo, fix integration bugs preventing the binary from working despite 400+ passing unit tests.

## Current State
Complete. All 411 tests pass, scan works (271 files, 1740 refs). Ready to commit.

## Problem/Context
Started from README rewrite session. Attempted to run sprefa on itself to test file renames. Hit three integration bugs that unit tests never exercised:
1. SQL reserved word collision in generated view DDL
2. Builtin extractor column name mismatch between table schema and INSERT
3. Config field name mismatch (branches vs revs)

## Solution/Approach
1. Quoted all dynamic column names/aliases in generated SQL (rule_tables.rs CREATE VIEW, CREATE TABLE; sqlite_store.rs INSERT)
2. Fixed extraction-to-row conversion: builtins (single capture where kind == rule_name) emit column "value" matching the table schema
3. Fixed sprefa.toml to use `revs` field

## Key Insights
- 422 unit tests passed while the binary couldn't scan a single file. Tests used in-memory fixtures that never hit the generated-SQL-against-real-SQLite path.
- The reserved word collision only triggers on user-defined capture names that happen to be SQL keywords ($TABLE, $ORDER, etc). The EXAMPLES.sprf had a rule capturing $TABLE.
- Builtin extractors (JS/TS, Rust) produce RawRefs where kind == rule_name (e.g. kind="rs_use", rule_name="rs_use"). The table schema used "value" as column name but insertion used the kind. Fix: detect builtins (single capture, kind == rule_name) and map to "value".
- The config struct has `revs: Option<Vec<String>>` but README/docs/add-command all said `branches`. The `add` command appends `branches = ["main"]` which silently gets ignored.

## Open Questions
- The `add` command hardcodes `branches = ["main"]` but the struct field is `revs`. Should rename the struct field or the add command output?
- Need an integration test that actually runs scan against a fixture repo
