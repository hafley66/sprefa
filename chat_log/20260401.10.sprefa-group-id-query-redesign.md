## Problem/Context
Started as planner chat designing anti-unification, schema review, drift detection. User promoted this chat to doer to implement rule-as-relation in the query compiler after seeing the doer chat hadn't wired it yet.

## Solution/Approach
Rule-as-relation: when relation_source encounters a name matching a known extraction rule, it generates SQL that JOINs matches within the same group_id, one JOIN per capture kind. Returns match IDs (not strings) so compile_final_select works unchanged. Built-in metadata relations return string values in all columns (all_string flag).

## Key Insights
- Rule relations return match IDs to reuse existing compile_final_select. All columns are match IDs joined via group_id.
- All-string builtins (repo_has_tag etc) return string values in ALL columns including a0. The is_string_col check handles literal matching correctly for both types.
- Variable unification between a rule-relation column (match ID) and an all-string builtin column (string) is a known gap -- they live in different domains. The negated builtin case works because NOT EXISTS just needs equality, but positive joins across these types would produce incorrect SQL.
- The 5 snapshot test failures are pre-existing from the group_id commit (group numbering changed), not from this session's changes.

## Open Questions
