# Session: sprefa-n1-blob-oid-implementation

## Goal
Implement the three-phase optimization plan from the prior session: eliminate N+1 queries, use git blob OIDs instead of mmap+xxh3 for skip checks, and replace all-or-nothing staleness with incremental file-level staleness.

## Current State
All three phases implemented and committed as db36d31. 381 tests pass. No remaining work items from the plan.

## Problem/Context
Four compounding perf bugs in the scan pipeline:
1. N+1 file inserts: flush_batch called ensure_file per-file (INSERT+SELECT each) + per-file rev_files INSERT
2. Unconditional mmap+hash: every file opened/hashed before skip check
3. Dead skip mechanism: scanner_hash never written to files table, skip_set always empty
4. All-or-nothing staleness: one stale file forced full rescan bail

## Solution/Approach
### Phase 2a+2b (blob OIDs + skip before mmap)
- files.rs: git2_list_files, list_files, diff_files, list_blobs_at_rev return (path, blob_oid) pairs. entry.id() is free from tree walk.
- extract.rs: extract_from_list accepts (PathBuf, String) pairs. Skip check runs BEFORE file open/mmap. extract_from_blobs uses blob OID as content_hash directly.
- Non-git walkdir fallback keeps mmap+xxh3 for hash.

### Phase 1a+1b (bulk inserts + scanner_hash)
- flush_batch gains scanner_hash param, writes it via ON CONFLICT DO UPDATE SET scanner_hash
- ensure_file deleted. Replaced with bulk INSERT chunks of FILE_CHUNK=2000, SELECT back IDs, bulk rev_files INSERT OR IGNORE.
- 1000 files: 3000 SQL roundtrips -> ~3

### Phase 3a+3b (incremental staleness)
- has_stale_scanner_hash(bool) replaced with stale_file_paths(Vec<String>) in Store trait + SqliteStore
- scan_diff merges stale paths into diff.changed with blob OIDs resolved from HEAD tree walk
- No more bail!("full rescan required")

## Key Insights
- git2::TreeEntry::id() returns blob SHA-1 without reading content -- already used for rename detection at files.rs:69, just never captured for skip checks
- Hash transition is clean: since scanner_hash was never written, skip_set was always empty. First scan with new code populates blob OIDs + scanner_hash. No migration needed.
- UNIQUE(repo_id, path, content_hash) constraint means old xxh3 rows persist as dead weight but don't interfere -- new blob OID rows get created alongside them
- Bulk INSERT + SELECT back pattern matches intern_strings approach already in the codebase
- The stale merge in scan_diff walks the HEAD tree filtering by a HashSet of stale paths to resolve their blob OIDs
