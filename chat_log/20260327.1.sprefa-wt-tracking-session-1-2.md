# Session: sprefa-wt-tracking-session-1-2

## Goal
Implement working-tree (+wt) branch tracking so the DB distinguishes committed state from local filesystem state. This session covers Sessions 1 and 2 of the 5-session plan at ~/.claude/plans/sleepy-tickling-popcorn.md.

## Current State
Sessions 1 and 2 complete. 269 tests passing, 0 failures. Changes uncommitted on master.

## Problem/Context
The daemon watches working copies and rewrites downstream files on rename. But the DB had no way to distinguish refs from committed scans vs working-tree mutations. Both wrote into the same branch_files under the same branch name. This caused: mixed query results, race conditions when cousin server rescans, inability to diff committed vs local state.

## Solution/Approach

### Session 1: Schema + branch helpers + flush
- Added `is_working_tree INTEGER NOT NULL DEFAULT 0` column to repo_branches (idempotent ALTER TABLE)
- Added `is_working_tree: bool` to RepoBranch struct
- Created `wt_branch()`, `is_wt_branch()`, `base_branch()` helpers in both cache/lib.rs and watch/lib.rs
- flush() now does full-replace DELETE+INSERT for +wt branches (complete on-disk state known), additive INSERT OR IGNORE for committed branches

### Session 2: Watcher branch awareness + daemon startup
- Added `wt_branch: Option<String>` to WatchConfig
- Create handler inserts file into files table + links to wt branch_files
- Delete handler removes from wt branch_files
- cmd_daemon scans main+wt on startup alongside committed main
- Extracted spawn_watchers() from duplicated cmd_watch/cmd_daemon loops

### Bonus: git checkout burst dedup
- macOS reports both Create and Modify for overwritten files during checkout
- Without dedup, Create handler inserts new file rows, then Modify handler hits FK constraint on the old rows
- Fix: before classification, check DB to resolve ambiguity. File exists in DB = modify. File not in DB = create. Remove the losing classification.

## Key Insights
- macOS FSEvents reports both Create and Modify for std::fs::write to existing files. The DB is the source of truth for which classification wins.
- wt branch helpers are duplicated in cache and watch to avoid adding cache as a watch dependency (potential circular dep). Trivial string ops so duplication is fine.
- flush full-replace for +wt is correct because the scanner walks all on-disk files. Committed branches stay additive because they accumulate across incremental fetches.
