## Problem/Context
1. Scan-order dep: resolve_match_links ran per-repo during scan, missing cross-repo links when target repo not yet indexed. Fixed with second pass in cmd_scan.

3. Current rule config is YAML/JSON with a predicate DSL. User wants to replace with a datalog-based language that unifies extraction rules, link rules, and queries into one syntax with bidirectionality.

## Solution/Approach
- Scan-order: second pass of resolve_links after all repos scanned (committed 9df8606)
- ghcache: optional dep + cfg(feature) guards (committed 545a1ea)
- Both repos pushed: hafley66/sprefa, hafley66/ghcacher
- Datalog DSL: explored design space thoroughly. Key decisions:
  - Datalog syntax (not CSS selectors) for bidirectionality
  - Pipe-delimited blocks for embedding regex/ast-grep patterns (d2-style)
  - Compiles to SQL (same backend as current link_compile.rs)
  - Same language for extraction rules, link rules, and queries
  - SQLite IS the datalog evaluator (INSERT OR IGNORE fixpoint = semi-naive)

## Key Insights
- The match_links table is the base relation. SQLite recursive CTEs + NOT EXISTS already provide datalog semantics (transitive closure, negation, disjunction) at query time. No need to add these to the rule compiler.
- Prolog/datalog bidirectionality avoids the two-mini-language problem (extraction vs linking as separate syntaxes). Everything is a relation.
- Pipe-delimited blocks (d2-style) solve the pattern embedding problem cleanly -- no backtick nesting, no regex escaping in strings.
- File path components as auto-emitted matches means the system has value before writing any extraction rules (convention detection, name-based linking).
- The ergonomic gap is a query interface, not more engine features. The engine is done.
- Semi-naive evaluation = INSERT OR IGNORE + fixpoint loop, which is exactly what resolve_match_links already does.
