## Problem/Context
Session 4 called for verifying the watcher pipeline works with the new matches-based schema (kind TEXT + rule_name TEXT instead of ref_kind INTEGER on refs). The watcher code (diff.rs, plan.rs, queries.rs, watcher.rs) was already migrated in prior sessions -- all queries JOIN through matches table, replace_file_refs inserts into matches, load_decl_refs reads from matches. The missing piece was an integration test proving end-to-end correctness with branch scoping.

Separately, session 5's query logging table was discussed and shelved -- SQLite has no built-in query log, and building application-level logging is premature without users. Instead, added a `sprefa sql` CLI command so users can query the DB directly without finding the file path.

## Solution/Approach

### Full pipeline integration test (crates/watch/tests/2_full_pipeline.rs)
4 tests using dual-branch scanning (main + main+wt):
1. js_rename_matches_table_and_branch_divergence -- JS re-export chain rename, SQL assertions on matches kinds/rule_names, verifies export_name and import_name matches with rule_name "js"
2. rs_rename_updates_matches_and_consumers -- Rust rename propagation, verifies rs_declare and rs_use matches with rule_name "rs". Key insight: rs_use stores full path (crate::utils::compute_score), not bare name.
3. branch_files_intact_after_rename_pipeline -- Dual-branch scan, rename, verifies no orphaned matches/refs, branch_files intact
4. new_file_during_watch_gets_indexed_with_matches -- New file during watch gets branch_files link, content change triggers re-extraction into refs + matches

### sprefa sql command (crates/cli/src/main.rs)
- `sprefa sql "<query>"` -- opens index DB auto-resolved from config, runs read-only query, prints tab-separated results with header
- Guards: only SELECT/WITH/EXPLAIN/PRAGMA allowed, semicolon-chained multi-statement injection blocked
- Column type-aware output (INTEGER/REAL/TEXT/NULL)

### DB restart
- Deleted old ~/.sprefa/index.db (had pre-migration schema), re-initialized, re-scanned. 22 refs with correct kinds/rule_names.

## Key Insights
- rs_use refs store full module paths (crate::utils::compute_score), not bare names. SQL queries for Rust refs need LIKE '%name%' or exact path match.
- SQLite has no built-in query log. sqlite3_trace_v2() exists at C API but sqlx doesn't expose it. Application-level logging is the only option for persistent query logs.
- SQLite triggers only fire on INSERT/UPDATE/DELETE, not SELECT -- can't use triggers for read logging.
- The existing DB at ~/.sprefa/index.db was from pre-migration sessions and had schema mismatches. Needed delete + reinit + rescan.
- Server wiring (query eval, cascade endpoint) shelved -- ghcacher library will handle GitHub integration via separate SQLite DB, CLI querying via `sprefa sql` is sufficient for now.
