# Session: sprf-lang-grammar-design

## Goal
Design the .sprf language grammar -- CSS-style selector chains with `>` combinator, JSON destructuring mini-lang, and unified capture syntax.

## Current State
Plan file finalized at ~/.claude/plans/drifting-stargazing-jellyfish.md. Grammar productions drafted and edge cases analyzed. Ready to implement but user wants more design iteration before code.

## Problem/Context
sprefa rules are currently JSON (sprefa-rules.json). Building a .sprf DSL that replaces it with a more expressive, readable syntax. The language has three sub-languages unified by one capture vocabulary: selector chain globs, JSON destructuring patterns, and ast-grep/regex embedded blocks.

## Solution/Approach
- `>` combinator separates pattern slots (CSS child combinator style)
- `<<...>>` delimiters for embedded patterns (ast-grep, regex, destructuring)
- ast-grep metavariable convention for captures everywhere:
  - `$VAR` = single match (like `*` in glob, one AST node)
  - `$$$VAR` = multi match (like `**` in glob, multiple AST nodes)
  - `$_` = wildcard, match but don't bind
  - Captures must be SCREAMING_CASE
  - `$lowercase` is just glob chars (not a capture)
- `$REPO` and `$BRANCH` are inherited context (CSS currentColor analogy)
- No keywords (no from/where/select)
- No literal mode -- everything is glob or regex
- Quotes only escape spaces and `>` for the parser
- `re:pattern` prefix for regex (since `/` is a path character)
- Whitespace-insensitive (no newline termination)
- `>>` close delimiter requires counting `>` pairs
- Destructuring engine: custom tree matcher on serde_json::Value, partial object matching, ancestor capture carry-forward through array iteration

## Key Insights
- ast-grep does exact structural matching on JSON -- can't do partial object match, so custom destructuring engine needed
- ast-grep confirmed working for JSON/YAML but NOT TOML
- serde_json::Value is universal tree repr for JSON/YAML/TOML -- destructuring engine operates on this
- Pattern AST doubles as match spec -- no intermediate representation
- Ancestor captures carry forward through array iteration (the killer feature over ast-grep for structured data)
- One capture vocabulary ($VAR/$$$VAR) across all three sub-languages unifies the whole thing
- `=` operator reserved as potential future constraint syntax ($VAR=pattern)
