# Session: sprefa-rev-slot-segment-captures

## Goal
Two related changes to the .sprf pattern system:
1. Merge branch() + tag() into rev() slot
2. Add segment captures and regex captures to all pattern positions (repo, rev, fs)

## Current State
- branch() and tag() are separate SelectStep variants and separate fields in CompiledGitSelector
- All pattern positions support glob and re: prefix, but matching is boolean only (no capture extraction)
- Context captures grab the whole value (ctx.branch, ctx.repo, path), not segments from the pattern
- PatternMatcher enum already has Glob and Regex arms in git_match.rs

## Plan

### Part 1: rev() slot

Rename across the stack:

| Layer | Before | After |
|-------|--------|-------|
| _0_ast.rs Tag enum | Branch, Tag (in sprf Tag enum, not git tag) | keep Fs/Json/Ast/Re/Repo, add Rev. Keep Branch as parse alias |
| _1_parse.rs | parses `branch(...)` and `tag(...)` separately | parse `rev(...)`, keep `branch()`/`tag()` as aliases |
| types.rs SelectStep | Branch { pattern, capture }, Tag { pattern, capture } | Rev { pattern, capture } |
| _3_lower.rs | lowers Branch/Tag to separate SelectStep variants | lowers all three keywords to SelectStep::Rev |
| git_match.rs CompiledGitSelector | branch: Vec<PatternMatcher>, tag: Vec<PatternMatcher> | rev: Vec<PatternMatcher> |
| git_match.rs matches() | checks branch and tag separately | checks rev against ctx.branch OR any ctx.tag |
| extractor.rs compile_rule | pushes to branch_patterns / tag_patterns | pushes to rev_patterns |
| extractor.rs ContextCaptureSource | Branch, Tag | Rev |
| extractor.rs resolve_context_captures | Branch -> ctx.branch, Tag -> ctx.tags[0] | Rev -> whichever matched (branch or first matching tag) |
| tmLanguage.json | no change needed, already highlights keywords generically |
| sprf-lsp completions | add "rev" to tag completions |

Files: _0_ast.rs, _1_parse.rs, _3_lower.rs, types.rs, git_match.rs, extractor.rs, sprf-lsp/main.rs, tmLanguage.json

### Part 2: segment captures in pattern positions

Add a third PatternMatcher variant for patterns containing $CAPTURES:

```rust
enum Segment {
    Literal(String),
    Capture(String),      // $X -- one path segment
    MultiCapture(String), // $$$X -- N segments, joined with /
    Wild,                 // $_ -- one segment, no bind
    MultiWild,            // $$$_ or ** -- N segments, no bind
}

enum PatternMatcher {
    Glob(GlobMatcher),
    Regex(Regex),
    SegmentCapture(Vec<Segment>),
}
```

Detection: pattern contains `$` -> segment capture, starts with `re:` -> regex, otherwise -> glob.

Add `captures()` method to PatternMatcher:
- Glob: returns None (no captures, whole value available via slot capture)
- Regex: extracts named groups (must be SCREAMING)
- SegmentCapture: splits input on `/`, walks segments, binds $VAR to matched segments

Wire into resolve_context_captures:
- Current: always grabs whole value from ctx
- New: if pattern has captures, extract them; whole value still available as implicit parent

Regex named groups must be SCREAMING ($CASE) -- enforced at parse time. `(?P<version>...)` is a parse error, `(?P<VERSION>...)` is valid.

Every capture has an implicit parent = the full matched value. If `repo($ORG/$REPO)` matches `acme/frontend`, captures are ORG=acme, REPO=frontend, and the slot-level capture (if any) is the full `acme/frontend`.

Files: git_match.rs (PatternMatcher + Segment type + match_segments fn), file_match.rs (same treatment for fs patterns), extractor.rs (resolve_context_captures upgrade)

### Implementation order
1. rev() rename first -- smaller, touches the same files, clears the path
2. Segment captures second -- builds on the simplified rev field

## Key Insights
- git calls branches + tags "refs" but sprefa already uses "refs" for the core DB table. "rev" avoids collision.
- Pipe-delimited OR already works within a single pattern via compile_patterns splitting on |. rev() gets branch|tag OR for free.
- The segment matcher is ~40 lines. One $$$VAR per pattern keeps it unambiguous.
- re: prefix already works in git_match.rs and json key patterns. Extending captures from regex named groups is wiring, not new logic.

## Open Questions
- Should branch() and tag() remain as parse aliases forever, or deprecate after migration?
- For fs() segment captures, does $NAME match one path component or one glob segment? (One path component, split on /)
- Should anti-unification output use segment captures or regex? (Segment captures for simple cases, regex for complex extraction)
- What about $_ in regex position? Not applicable -- regex has its own wildcard syntax.
