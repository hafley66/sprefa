# Session: sprefa-scaffold-rules-engine

## Goal
Build sprefa (super-refactor) -- a daemon-powered rename refactoring tool that indexes source files from N git repos into SQLite, extracts every interesting string as a ref with byte-level spans, and enables instant deterministic rename propagation without LLMs.

## Problem/Context
User built a cross-repo code index at work (Rust + SQLite) and wants to genericize it as an open-source tool. The end goal is NOT just indexing -- it's a daemon that watches project folders and performs instant rename refactoring across all files/imports/exports. The index is the foundation; rename is the product.

Key pain point at work: the ways files reference each other (deps, imports, config keys, service URLs in YAML) keep changing. Writing Rust for each new pattern is unsustainable.

## Solution/Approach
### Rule engine (next to build)
A JSON-schema-validated rule language for declaring "how strings in structured files point to things in other repos." Three selector dimensions:

1. **Git context**: repo glob, branch glob, tag glob (main vs LTS branches need different rules)
2. **File path**: glob on file path within repo
3. **Structural position**: CSS/jq hybrid path selectors inside parsed files, plus ast-grep patterns for code files

Path selector primitives: `any` (descendant **), `key` (exact), `key_match` (glob on key name), `index` (array), `depth_min/max/exact`, with ast-grep fallback via `select_ast` for code files.

Execution: git context match -> file glob match -> parse -> structural select -> value pattern capture -> emit RawRef

### Parser strategy (Phase 2)
- oxc_parser for JS/TS (MVP language)
- tree-sitter + ast-grep as catchall
- Custom JSON/YAML/TOML with lodash-style dot-path key encoding
- SCIP index consumption for high-fidelity symbol resolution when available

### Architecture
- Hybrid: own SQLite index for fast normalized string matching + SCIP consumption for semantic accuracy
- Daemon mode: filesystem watcher, keeps index hot, serves rename operations
- CLI mode: same transaction logic, direct SQLite access

## Key Insights
- The product is rename refactoring, not code search. The index exists so renames are O(lookup).
- "Most renames are unambiguous string matches within a known module graph" -- you don't need full type inference.
- SCIP is the interchange format to consume (rust-analyzer emits it, `scip` crate on crates.io).
- ctags is definitions-only, insufficient. ra_ap_ide is too heavy to embed.
- The rule engine is the key differentiator: declarative rules for how arbitrary structured files reference each other, without recompiling.
- User hates: #[path] renames, Cargo's - to _ normalization, over-abstraction, LLMs doing mechanical refactoring.
