# Session: sprefa-schema-migration-plan

## Goal
Design the next major phase of sprefa: migrate ref_kind from a hardcoded Rust enum on the refs table to user-definable text strings on a matches table, separating physical refs from semantic interpretations.

## Current State
Plan approved, not yet implemented. GitHub repo set up at hafley66/sprefa (SSH remote configured). All prior work (sessions 1-4 of +wt plan, query crate, extractors) is committed on master.

## Problem/Context
Three design discussions happened this session:

### Discussion 1: What is sprefa?
User reframed the vision: sprefa is a Prolog for codebases. Extraction rules are declarative config that point the algorithm at what to index. It captures ALL string refs across languages -- identifiers, FS paths, AST paths, config values, env vars, git tags, URL strings, operationIds from OpenAPI that codegen repeats into function names. The matching works across filesystem (globs), ASTs (extractors), JSON/YAML/TOML paths (tree walk), and branches (time). The 500-repo use case: deploy/k8s configs, cross-language codegen, version strings, cross-repo package refs.

### Discussion 2: Query crate disposition
The query crate (Expr tree, evaluator, standing rules) was built speculatively. User's position: SQLite IS the query interface. FTS5 trigram, JOINs, normalized columns -- that's the query language. No abstraction over SQL. The query crate sits dormant. Real query patterns will emerge from usage against the DB, then we can decide what (if anything) to abstract. "I love SQL, I abhor re-inventing it."

### Discussion 3: Schema redesign
Walked through ref_kind limitations (hardcoded enum, can't add kinds from config) and match cardinality (what happens when two rules find the same string at the same byte offset). Arrived at physical/semantic separation:

- **refs** = physical: "string X at byte Y in file Z". UNIQUE(file_id, string_id, span_start). No ref_kind column.
- **matches** = semantic: "rule R says ref X is kind K". UNIQUE(ref_id, rule_name). Kind is free text.
- **match_labels** = metadata on matches: key/value pairs. UNIQUE(match_id, key).

Language extractors write rule_name="js"/"rs" with known kind strings. User rules write whatever kind string they define in config. Everything queryable with WHERE kind = 'whatever'.

## Solution/Approach
5-session plan written to ~/.claude/plans/sleepy-tickling-popcorn.md:
1. Schema migration -- ref_kind moves from refs to matches_v2, all insert paths updated
2. Extraction rule coverage -- new rules for tsconfig, deploy, k8s, openapi, workspace configs using user-defined kinds
3. End-to-end extraction test -- multi-repo fixture, raw SQL assertions
4. Watcher pipeline update -- JOINs through matches_v2 for kind info, integration test
5. DB documentation + query logging table

## Key Insights
1. A ref is physical (string at byte offset in file). All semantic meaning (kind, rule, labels) belongs on the match. One ref, multiple interpretations.
2. SQLite is the query protocol. No Expr abstraction needed. Query patterns emerge from usage.
3. User-defined kind strings from config eliminate the RefKind enum bottleneck. Adding a new extraction pattern is pure config, no Rust changes.
4. The authoring maturity ladder for any future query DSL: Rust API first, then YAML serialization, then text language, CLI last.
5. The query crate is dormant, not deleted. It may inform future abstractions once real query patterns accumulate.
