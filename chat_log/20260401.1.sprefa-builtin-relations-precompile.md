# Session: sprefa-builtin-relations-precompile

## Goal
Three features in one session:
1. Pre-compile walk step patterns (eliminate N+1 recompilation per node)
2. Topological sorting + cycle detection for query dependencies
3. Built-in relations for the query compiler (prolog-style unification over base tables)

## Current State
All three implemented, all tests pass, nothing committed yet.

### Pre-compiled walk steps
- New `CompiledStep` enum in walk.rs mirrors SelectStep but holds pre-built `Vec<PatternMatcher>`
- `compile_steps()` called once during `compile_rule` in extractor.rs
- `walk()` takes `&[CompiledStep]`, `walk_select()` convenience wrapper for tests
- CompiledRule.steps changed from `Vec<SelectStep>` to `Vec<CompiledStep>`
- Also compiled: `CompiledKeyMatcher` for Object entry glob patterns, `CompiledObjectEntry`

### Query dependency resolution
- `compile_query_with_deps()` topologically sorts all transitive query dependencies
- Emits chained CTEs in correct order so dependent queries are defined before referenced
- Cycle detection for non-self-referencing cycles (self-refs are legit WITH RECURSIVE)
- CLI cmd_query_datalog updated to use compile_query_with_deps

### Built-in relations (8 total)
Added to `relation_source()` via `builtin_relation()`:
- `has_kind($M, "kind")` -- match has this kind
- `has_norm($M, "val")` -- match string normalized value
- `has_value($M, "val")` -- match raw string value
- `same_norm($A, $B)` -- two matches share same norm
- `same_repo($A, $B)` -- two matches in same repo
- `same_file($A, $B)` -- two matches in same file
- `in_repo($M, "name")` -- match is in named repo
- `in_file($M, "path")` -- match is in file at path

Literal handling: built-in relation non-a0 columns expose string values directly,
so literal args use `= 'value'` instead of the match-ID-to-string subquery resolution
used by link_kind relations.

## Key Insights
- Built-in relations turn the query language into a real relational query over the full schema, not just a graph walk over pre-materialized link edges
- Link rules become an optimization (materialize hot paths) rather than a requirement
- Variable sharing across atoms IS unification: $X in two atoms compiles to SQL JOIN ON
- WITH RECURSIVE handles transitive closure, SQLite caps at 1000 depth
- same_org not needed when all repos are same org (user confirmed)
- The deploy->repo->package-lock->peer-dep chain is expressible as 3 query rules with no link rules needed
