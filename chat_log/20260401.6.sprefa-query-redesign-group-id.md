# Session: sprefa-query-redesign-group-id

## Goal
Design the next major evolution of the .sprf query layer: group_id on matches, rule-as-relation, check construct with negation, and unified syntax.

## Current State
Design phase complete. Plan file written at ~/.claude/plans/crystalline-finding-dolphin.md. No code changes made -- this was a planner chat, doer chat is separate.

## Problem/Context
The .sprf query language operates on match IDs (opaque integers in matches table), causing three problems:
1. Context re-derivation: every query re-surfaces repo/file/rev context via builtins (in_repo, same_file, has_kind) even though this context already exists in the schema
2. Floating matches: captures from same extraction site become independent rows in matches, losing co-extraction context
3. No invariant checking: no way to declare "this must hold" for drift detection

Additional discussion covered:
- Anti-unification algorithm for inferring .sprf patterns from two file references (whiteboard stage)
- SQLite schema review: refs vs repo_refs overlap, branch/tag table unification, normalization decisions
- Whether current primitives (strings + SQL) cover ambitions for type trees, call graphs, JSX trees, cross-repo/cross-process rules (answer: yes, gap is always in extraction not schema)
- IS_REPO/IS_REV demand scanning as the mechanism that makes cross-repo rules practical
- Commit graph as uncharted territory (deliberately left alone -- no use case)

## Solution/Approach
Unified syntax: `keyword name(args) > body;` for all four statement types (rule, link, query, check).

Key design decisions:
1. **group_id** on matches table ties co-extracted captures together
2. **Rule name = queryable relation** with captures as columns, string-valued variables
3. **Rule head declarations** replace trailing match() slots: `rule deploy_config($SVC, repo($REPO), rev($TAG)) > selectors;`
4. **repo()/rev()** annotations on captures for scan roles (replace IS_REPO/IS_REV)
5. **check** = query whose non-empty results are violations, `not` prefix for negation (NOT EXISTS)
6. **Built-in metadata relations** (repo_has_tag, repo_has_branch) over base tables
7. **Links** as materialization directives with prefix-call predicates and dot notation: `link dep_match($A, $B) > norm_eq(rule_a.$CAP($A), rule_b.$CAP($B));`
8. **Rule unions**: same name = same relation, shape-checked at compile time. No variant syntax -- if you care which def produced a row, use different names.
9. **Match kinds gone from surface syntax**: kind = lowercased variable name from head, scoped by rule_name
10. **Phase separation**: > chains for extraction (ordered), atom bodies for queries (unordered)

## Key Insights
- The string+SQL thesis is sufficient for type trees, call graphs, JSX trees, cross-repo rules. The gap is always extraction, never schema.
- repo_refs table is not duplication -- it's the adapter between structured metadata tables and the string interning layer. Load-bearing for match_links.
- IS_REPO/IS_REV demand scanning closes the loop on cross-repo coverage without upfront enumeration. Rules declare the scan frontier.
- Match kinds as a global namespace was a "shotgun blast". Scoping captures to rules (rule_name.kind) keeps co-extracted values coupled.
- Anti-unification over (path, data tree, AST, string) layers could auto-generate .sprf patterns from examples -- whiteboard stage, not planned for implementation.
- User preference: no Prolog syntax (no :-, no \+, no commas). Use `not` for negation, newline-delimited body atoms.
- Rule union disambiguation: just use different names. No variant/index syntax needed.
