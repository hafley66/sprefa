## Rust vs TypeScript for Scope Context

The user asked about implementing scope context in Rust vs TypeScript:

### Key Differences

**1. Borrow Checker vs Nested Contexts**
- TS: `child.parent = parentScope` just works (references)
- Rust: References require lifetimes ('a) - painful for trees
- Solution: Use arena pattern - `Vec<Scope>` with usize parent indices

**2. Clone vs Reference**
- TS: Object references are cheap
- Rust: Cloning HashMaps for each scope is expensive
- Solution: Stack of maps with parent lookup, or persistent data structures

**3. Enums with Data (Sum Types)**
- TS: Unions are structural, runtime checks
- Rust: Enums are tagged, exhaustive match checks at compile time
- Benefit: Can't forget to handle Block case

**4. String Management**
- TS: All strings are the same
- Rust: String (owned) vs &str (borrowed slice)
- Trap: Can't store &str into context if source text gets dropped
- Solution: Use String or Arc<str> for context values

### Implementation Pattern for RuleBody

```rust
// Arena-based scope tree
struct ScopeArena {
    scopes: Vec<Scope>,
}

struct Scope {
    vars: HashMap<String, String>,
    parent: Option<usize>, // index into scopes Vec
}

// Recursive rule body
enum RuleBody {
    Step(Slot), // A single match
    Block {
        scope_slot: Slot, // e.g., folder(packages/$PACKAGE)
        children: Vec<RuleBody>,
    },
}
```

### Lookup Strategy

```rust
fn resolve_var(&self, scope_id: usize, var: &str) -> Option<&str> {
    let scope = &self.scopes[scope_id];
    if let Some(val) = scope.vars.get(var) {
        return Some(val);
    }
    // Walk up parent chain
    if let Some(parent_id) = scope.parent {
        return self.resolve_var(parent_id, var);
    }
    None
}
```

This mirrors JS closures but with explicit parent pointers instead of references.
