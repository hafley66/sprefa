## Solution/Approach

- Git selector now actually checked at extraction time in RuleExtractor (was compiled but never evaluated).

- All 268 tests pass, 3 snapshots updated (schema, minimal_rule, full_rule_with_captures).

### Graph model (docs/graph-model.md)
- Index modeled as typed DOM tree with graph edge overlay
- Built-in tables (repos, files, refs, matches) keep typed columns and indexes
- User-defined nodes/edges in generic normalized tables (custom_nodes + custom_node_attrs, edges + edge_attrs)
- Element trait unifies both for CSS selector engine
- Four rule primitives: emit, link, spawn, crawl
- Stratified Datalog reduction: emit=simple rule, link=join, spawn=negated rule, crawl=cross-stratum trigger
- CSS selectors compile to conjunctive goals over materialized fact base
- Pipeline: extract → link/spawn (fixed point) → crawl → repeat

## Key Insights
- ExtractContext uses borrowed refs with lifetime to avoid cloning repo/branch strings on every extract call
- The Action wrapper existed to hold target_repo/target_path/confidence alongside emit. With those removed (dead code), emit goes directly on Rule.
- Git selector was compiled into CompiledGitSelector but never checked because Extractor::extract had no git context parameter. Now wired.
- The graph model reduces to stratified Datalog: four YAML rule clauses map to four kinds of Horn clauses. CSS selectors are conjunctive queries.
- Built-in typed tables vs generic tables is a storage optimization. In the logic model they're all just predicates with different arities.
- spawn uses negation-as-failure (only create entity if link found nothing), which requires stratification: all links must complete before spawn checks absence.
