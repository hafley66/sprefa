# Session: sprefa-unified-syntax-rule-heads

## Goal
Align the .sprf parser and query compiler with the planned language design from the crystalline-finding-dolphin plan. The prior doer sessions had implemented backend infrastructure (group_id, rule-as-relation SQL, check/negation types) but went wrong on surface syntax (Prolog-style :- with commas instead of > with whitespace-delimited bodies). Rule head declarations were never started.

## Current State
All three syntax changes landed and committed as 7f504d7:

1. query/check delimiter: :- replaced with >, commas replaced with whitespace-delimited body atoms
2. rule head declarations: `rule name($VAR, repo($VAR), rev($VAR)) > selectors;` replaces trailing match() slots
3. string-valued rule-as-relation: rule_relation_source emits strings, compile_final_select splits into string vs match-ID paths

130 tests pass (60 sprf + 70 rules). 5 pre-existing snapshot failures from group_id numbering. Full pipeline verified: repo()/rev() annotations flow through extraction -> match_labels -> discovery -> scan_rev().

## Problem/Context
User noticed after 2 rounds of doer sessions that the parser went the wrong direction -- implemented Prolog :- with commas when the plan called for > with newline-delimited bodies. Rule head syntax (the biggest surface change) was never touched. User said "unfuck and get back on track, one step at a time".

Also fixed the variable unification gap: rule-relations emitted match IDs while all-string builtins emitted strings, making positive joins across them produce wrong SQL. Now rule-relations resolve to strings, enabling direct unification.

## Solution/Approach
Three incremental steps:
- Step 1: Replace :- with > and commas with whitespace in query/check parser
- Step 2: Add rule head syntax (RuleDecl, Capture, CaptureAnnotation), remove Slot::Match/ScanRole/Tag::Rule
- Step 3: Make rule_relation_source emit string values, add is_all_string_relation, split compile_final_select

## Key Insights
- The is_string_col logic for literal matching needs to distinguish builtins (a0=match ID, a1+=string) from match_links (all match IDs) from rule-relations (all strings). The old `col_idx > 0` shortcut only applies to builtins.
- compile_final_select detects all-string queries by checking if all positive body atoms are string-valued (rule-relations, all-string builtins, or self-references). All-string queries skip the match ID resolution JOINs entirely.
- Discovery pipeline is fully wired: repo()/rev() -> CaptureAnnotation -> MatchDef.scan -> RawRef.scan -> match_labels -> discover_scan_targets() -> scan_rev()

## Open Questions
- Link syntax needs redesign since kinds are now capture variable names scoped to rules (plan mentions dot notation)
- Whether to keep backward compat for headless rules or require rule keyword everywhere
