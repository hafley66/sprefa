## Problem/Context
The daemon watcher pipeline had three bugs preventing live rename propagation:

1. **DECL_KINDS missing ImportName** in watcher.rs -- `load_decl_refs` SQL only bound 3 of 4 declaration kinds via hardcoded `IN (?, ?, ?)`. Made dynamic with loop over DECL_KINDS array.

2. **UNIQUE(file_id, string_id, span_start) on refs table** -- re-export spans produce both ExportName and ImportName at the same position. `INSERT OR IGNORE` silently dropped the second ref. Fixed by adding `ref_kind` to the UNIQUE constraint. Required fresh DB (migration was too fragile for the run-every-startup system, so CREATE TABLE was updated and users delete+rescan).

3. **FK constraint on update_content_hash** -- deleting duplicate file rows without first deleting their refs caused FOREIGN KEY failures. Added refs deletion before file deletion.

After fixes: full pipeline works end-to-end. Rename in utils.ts propagates through barrel.ts re-export chain to consumer.ts. Second-pass re-entry terminates cleanly (0 edits).

## Key Insights
- macOS `sed -i ''` produces Delete+Create events (not Modify). Direct `std::fs::write` produces Modify events. Both paths work in the watcher now.
- tempdir paths on macOS need `.canonicalize()` because `/tmp` -> `/private/var/folders/...` symlink resolution. Without this, watcher's rel_path doesn't match DB paths.
- The re-entry loop (rewrite triggers new notify events) terminates naturally because diff finds old_name == new_name on the second pass. No explicit cycle detection needed.
- `try_send` on the notify->debounce mpsc(512) is the backpressure valve. Dropped events during git checkout bursts require a rescan, which is the planned behavior.
- Rule-based extraction and refactoring coexist in the same refs table, distinguished by ref_kind. The UNIQUE fix (adding ref_kind) was necessary for both systems.

## Open Questions
- Should the migration system get proper version tracking instead of run-every-startup idempotent statements? The ALTER TABLE approach for schema changes is fragile.
- Usage-site references (call sites within function bodies) are the next static analysis frontier. oxc_semantic has scope analysis that could track all Identifier nodes resolving to an import binding.
- Task #5 (bundler alias resolution) was canceled by user -- tsconfig path aliases via oxc_resolver are sufficient.
