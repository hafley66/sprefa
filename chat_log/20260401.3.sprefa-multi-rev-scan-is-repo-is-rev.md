# Session: sprefa-multi-rev-scan-is-repo-is-rev

## Goal
Enable scanning file trees at arbitrary git revisions (tags, branches) without checkout, driven by two mechanisms:
1. Static (`scan_tags` config on RepoConfig): scan all tags matching glob patterns via git2 blob reader
2. Dynamic (IS_REPO/IS_REV match annotations): extraction rules mark captures as scan targets, scanner collects (repo, rev) pairs and chases them in a fixed-point loop

## Problem/Context
Scanner reads files from disk (mmap), only indexes HEAD. Tags stored as repo_refs (string metadata) but file trees at those tags never walked. Query layer can connect deploy chart image tags to repo git_tags via same_norm, but can't answer "what deps does repo X have at tag v2.3.1?"

Two-tier design: static scan (config-driven tag scanning) + dynamic scan (match annotations drive discovery loop).

## Solution/Approach
### IS_REPO/IS_REV annotations
- .sprf syntax: `match($REPO, image_repo, IS_REPO)` / `match($TAG, image_tag, IS_REV)`
- Threads through: AST -> parser -> lower -> MatchDef -> RawRef -> emit -> flush
- Flush writes to existing match_labels table (key="scan", value="repo"|"rev")
- Discovery loop queries match_labels, joins through parent_key to pair repo+rev matches

### Git blob reader
- list_blobs_at_rev(): git2 revparse -> tree walk -> blob.content() -- no checkout needed
- extract_from_blobs(): same as extract_from_list but from blob bytes not mmap
- Filesystem path kept for working tree (uncommitted changes), blob path for historical revs

### Discovery loop
- After tier-1 scans + link resolution, query match_labels for (repo, rev) pairs
- Filter: repo in config? rev already scanned?
- scan_rev() for new targets, resolve_links(), repeat until stable (max 10 iterations)
- discovery_log table records each target and its disposition

## Key Insights
- Filesystem path (mmap) exists for working tree uncommitted changes, not just performance. git2 blobs can only read committed content. Both paths are needed.
- IS_REPO/IS_REV is not redundant with match kind -- kind is freeform ("image_repo", "dep_name", etc), scan role is a cross-cutting annotation saying "this value is a scan target component." Multiple kinds can have IS_REPO.
- match_labels table already exists in schema (line 167), never written to. No migration needed for scan labels.
- branch_files junction accepts arbitrary branch strings -- tag names work as branch keys. No schema change needed.
- Parent field on MatchDef pairs repo+rev matches for discovery query joins.
- Discovery is a prolog fact emission -- IS_REPO/IS_REV emits new facts to match_labels which trigger lazy side effects (scanning).

## Open Questions
