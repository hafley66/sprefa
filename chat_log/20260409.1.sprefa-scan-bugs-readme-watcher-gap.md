## Key Insights
- The watcher's Rust rewrite path handles `use crate::X::Y` in consumer files but not `mod X;` in the module's own lib.rs. `mod` declarations are extracted as `rs_mod` kind but the rewrite planner doesn't map them back to file paths for rename propagation.
- `pub use _0_types as types;` re-exports don't work for internal `crate::types::` references within the same crate. Rust's module resolution doesn't follow `pub use X as Y` for intra-crate `use crate::Y::Item` paths.
- The `add` CLI command hardcodes `branches = ["main"]` in the TOML output but the config struct field is `revs`. This means every `sprefa add` produces a broken config entry.
- User feedback: blanket ban on negation words in descriptive text. Do not use "not", "no", "without", "never", "don't" etc. Rephrase positively.

## Open Questions
- Should `mod` declaration rewriting be a new phase in the rewrite planner, or should the Rust extractor emit a different ref kind for mod declarations?
- Should internal crate paths be handled by generating re-exports (lib.rs `pub use X as old_name`) or by rewriting all internal references?
- The `add` command's hardcoded `branches = ["main"]` needs to either use the correct field name `revs` or the struct needs a `branches` alias
