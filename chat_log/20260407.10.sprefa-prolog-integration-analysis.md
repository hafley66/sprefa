# Prolog Integration Analysis for sprefa

**Date:** 2026-04-07  
**Context:** User asked about scoping rules and whether current design boxes out future Prolog integration

## Executive Summary

**Key insight:** sprefa's current architecture (extraction → per-rule tables) is **COMPATIBLE** with Prolog/Datalog query layers. You're designing yourself **INTO** a clean two-layer architecture, not out of Prolog.

**Recommendation:** Keep sprefa as the deterministic extraction layer, add Datalog/Prolog query layer on top of per-rule tables. This separation of concerns makes Prolog integration **easier**, not harder.

---

## Current Scoping Model in sprefa

### 1. Lexical/Block Scoping

```sprf
rule(deploy) {
    repo($REPO) {           // ← Block: $REPO captured here
        rev(main) {         // ← Nested block: $REPO still visible
            fs(**/values.yaml) > json({ svc: $SVC })
                              // ← $REPO, $SVC both visible here
        }
    }
};
```

**Implementation:**
- `RuleBody::Block { slot, children }` in `crates/sprf/src/_0_ast.rs:62`
- Variables captured in `slot` flow into `children`
- Flattened to `ScopedStep` with `depth` and `scope_vars: HashSet<String>` (`_3_lower.rs:82-88`)
- Sorted by depth: outer scopes execute before inner (`_3_lower.rs:102-103`)

**Semantics:** One-way binding: extract → bind → pass downstream

---

### 2. Cross-Rule Scoping

```sprf
rule(deploy_config) {
    fs(**/services.yaml) > json({ repo: repo($REPO), pin: rev($PIN) })
};

rule(svc_version) {
    deploy_config(repo: $REPO, pin: $PIN)   // ← Binds columns from deploy_config
    repo($REPO) > rev($PIN) > fs(**/package.json)
};
```

**Implementation:**
- `RuleBody::Ref { cross_ref, children }` in `_0_ast.rs:68-73`
- Creates `DepEdge { producer, consumer, bindings }` (`_3_lower.rs:55-80`)
- DAG execution: producer runs before consumer

**Semantics:** Still one-way: producer output → consumer input

---

### 3. Variable Syntax

- `$SCREAMING_CASE` = capture variable (e.g., `$REPO`, `$TAG`)
- `$_` = wildcard (match but don't bind)
- `${NAME}` = braced capture for adjacent literals (e.g., `use${ENTITY}Query`)
- Extracted via regex (`_0_ast.rs:202-237`), stored as strings in SQLite

## Compatibility Matrix

### What's Compatible with Prolog ✅

| sprefa Feature | Prolog Analog | Status |
|----------------|---------------|--------|
| `rule(name) { body }` | Horn clauses | ✅ Similar structure |
| `$VAR` captures | Logical variables | ⚠️ One-way only |
| `other_rule(col: $VAR)` | Predicate calls | ✅ Similar semantics |
| DAG execution | Dependency-directed evaluation | ✅ Compatible |
| Per-rule tables | Extensional database (EDB) | ✅ Already have it |

### What's Missing (Prolog features you don't have) ❌

| Prolog Feature | Current sprefa | Gap |
|----------------|----------------|-----|
| **Unification** | One-way extract→bind | ❌ No bidirectional matching |
| **Backtracking** | Single-pass, no choice points | ❌ Failure = rule fails |
| **Logical variables** | Extract-only | ❌ Can't be input OR output |
| **Negation as failure** (`\+`) | No failure-driven control | ❌ No `\+ goal` |
| **Recursion** | DAG (no cycles) | ❌ Rules can't call themselves |
| **Disjunction** (`;`) | No alternatives | ❌ No `goal1 ; goal2` |

## The Fundamental Tension

**sprefa is a dataflow extraction engine**, not a **relational query engine**:

```
sprefa:  text → extract → bind → store → done
Prolog:  goal → unify → (fail? backtrack) → succeed → bind
```

**Your current model:**
```
1. Match file pattern
2. Extract captures (one-way: text → $VAR)
3. Store in per-rule table
4. Next rule reads from table
```

**Prolog model:**
```
1. Try to unify goal with facts/rules
2. If fail → backtrack to last choice point
3. If succeed → bind variables (could be input OR output)
4. Continue with next goal (failure-driven loop)
```

## Three Design Paths Forward

### Path 1: Two-Layer Architecture (Recommended ✅)

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: sprefa (extraction)                      │
│  .sprf rules → extract → per-rule SQLite tables   │
│  Deterministic, single-pass, DAG-ordered          │
└─────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│  Layer 2: Query engine (Prolog/Datalog/SQL)        │
│  Query per-rule tables with unification/backtracking│
│  Relational, bidirectional, recursive              │
└─────────────────────────────────────────────────────┘
```

**Example workflow:**

```sprf
// Layer 1: sprefa extraction (unchanged)
rule(container_images) {
    fs(**/values.yaml) > json({ image: repo($REPO) + ":" + rev($TAG) })
};

rule(packages) {
    fs(**/package.json) > json({ name: $NAME, version: $VER })
};
```

```prolog
/* Layer 2: Prolog query over extracted tables */
/* Find all services using outdated versions */
outdated_service(Svc, UsedVer, LatestVer) :-
    container_images(REPO, TAG),
    packages(NAME, UsedVer),
    REPO = NAME,
    semver_compare(UsedVer, LatestVer, lt),
    Svc = NAME.
```

**Why this works:**
- ✅ Per-rule tables are already relational (fast JOINs)
- ✅ Extraction stays deterministic (no backtracking overhead)
- ✅ Prolog/Datalog layer can be added later without breaking sprefa
- ✅ You can use existing Datalog engines (datafrog, crepe, ascent) or embed Prolog

---

### Path 2: Hybrid .sprf with `check` Blocks

Add Prolog-like validation as post-extraction:

```sprf
rule(container_images) {
    fs(**/values.yaml) > json({ image: repo($REPO) + ":" + rev($TAG) })
};

check {
    /* SQL-like or Prolog-like goals */
    container_images($REPO, $TAG),
    packages($REPO, $TAG),  /* Same repo+tag must have package.json */
    ensure($TAG != "latest")
};
```

**Execution:**
1. Run all `rule()` blocks (extraction)
2. Run all `check` blocks (validation)
3. Fail if any check fails

**Pros:**
- ✅ Keeps extraction simple
- ✅ Adds declarative validation
- ✅ Can compile `check` to SQL or Datalog

**Cons:**
- ⚠️ Still no backtracking in extraction
- ⚠️ Limited Prolog features (no recursion in checks?)

---

### Path 3: Full Prolog Embedding (Hardest ⚠️)

Redefine .sprf semantics around unification:

```sprf
rule(deploy_config(REPO, TAG)) :-
    fs(**/services.yaml),
    json({ repo: REPO, pin: TAG }).

rule(svc_version(REPO, TAG, VERSION)) :-
    deploy_config(REPO, TAG),
    repo(REPO),
    rev(TAG),
    fs(**/package.json),
    json({ version: VERSION }).
```

**What breaks:**
- ❌ Current `RuleBody` AST would need complete rewrite
- ❌ Execution model needs choice points, backtracking
- ❌ Per-rule tables assume single-pass, not failure-driven
- ❌ DAG execution prevents recursive rules

**What you'd gain:**
- ✅ True bidirectional queries
- ✅ Backtracking search
- ✅ Recursive rules

## Recommendation

**Don't fight it.** You've accidentally designed the **right architecture**:

1. **Keep sprefa as extraction layer** (deterministic, fast, DAG-ordered)
2. **Add Datalog/Prolog query layer on top** (relational, bidirectional)
3. **Use per-rule tables as the interface** (already relational!)

**Why this is the right call:**

| Concern | Extraction Layer | Query Layer |
|---------|-----------------|-------------|
| Performance | Single-pass, no backtracking overhead | Optimized joins (SQLite/Datalog) |
| Semantics | Deterministic, predictable | Declarative, relational |
| Complexity | Simple: match→extract→store | Rich: unification, recursion |
| Debugging | Easy: see what each rule extracted | Easy: query results, trace proofs |

**You're not designing yourself out of Prolog** — you're designing yourself into a **clean separation of concerns** that makes Prolog integration **easier**, not harder.

## Datalog-in-Rust Options

If you want to add a query layer, here are the main Rust-embedded options:

### datafrog
- **Used by:** Polonius (Rust borrow checker)
- **Evaluation:** Bottom-up, semi-naive
- **Integration:** Rust library, compiles to native code
- **Best for:** Large-scale program analysis, millions of facts

### crepe
- **Syntax:** Proc-macro Datalog (looks like Prolog)
- **Evaluation:** Bottom-up
- **Integration:** Compile-time Datalog → Rust
- **Best for:** Embedded queries, compile-time guarantees

### ascent
- **Features:** Lattices, fixpoint iteration
- **Evaluation:** Semi-naive with optimizations
- **Integration:** Macro-based, generates Rust
- **Best for:** Complex aggregations, numerical domains

### Comparison with sprefa needs

| Feature | datafrog | crepe | ascent | sprefa needs |
|---------|----------|-------|--------|--------------|
| Bottom-up eval | ✅ | ✅ | ✅ | ✅ (DAG execution) |
| Stratified negation | ✅ | ✅ | ✅ | ✅ (check blocks) |
| Aggregation | ❌ | ❌ | ✅ | ⚠️ (semver, counts) |
| Rust embedding | ✅ | ✅ | ✅ | ✅ |
| SQLite integration | Manual | Manual | Manual | ✅ (per-rule tables) |

**Recommendation:** Start with **crepe** for simplicity, or **ascent** if you need aggregation/lattices.

## Questions to Consider

1. **What Prolog features do you actually want?**
   - Bidirectional queries? (Datalog can do this)
   - Backtracking search? (Rarely needed for code queries)
   - Recursion? (Datalog has it, SQL has CTEs)
   - Negation as failure? (Datalog has stratified negation)

2. **Are you imagining `.sprf` files that look like Prolog**, or **Prolog queries that query sprefa's tables**?

3. **Do you need recursion in extraction**, or just in queries?
   - Extraction: DAG is fine (no recursive rules needed)
   - Queries: Datalog/Prolog layer can handle recursion
