# Session: sprefa-datalog-query-layer

## Goal
Add Datalog-style query rules to .sprf DSL for recursive graph traversal over match_links.

## Current State
Query layer is built and committed (3f04452). Full pipeline works: parse horn clauses in .sprf, lower to QueryDef, compile to SQL recursive CTEs, execute via `sprefa query 'goal($VAR, "literal")'`. LSP validates query rules with two-pass ordering. tmLanguage highlights query syntax.

Also added `same_file` link predicate and `package_has_dep` link rule to enable recursive chaining (unstaged).

## Problem/Context
match_links forms a graph but had no structured query interface beyond raw SQL. Link rules produce single-hop joins. Needed recursive traversal for transitive dependency chains, shared ecosystem queries, and cross-repo graph walks.

Key realization mid-session: recursion over a single link kind is useless because links connect different match kinds (dep_name -> package_name). The target of one link is never the source of the same link type. Recursion requires two link kinds per step: one to cross boundaries (dep_to_package, norm_eq) and one to walk within a file (package_has_dep, same_file).

## Solution/Approach
- `query` keyword in .sprf with Datalog horn clause syntax: `query head($A, $C) :- body($A, $B), head($B, $C);`
- Compiles to SQL WITH RECURSIVE CTEs. Variable unification = JOIN ON conditions.
- Link vs query semantic split: links are materialized edges (INSERT into match_links), queries are virtual views (SELECT with CTE). Two keywords, two concerns.
- `sprefa query` CLI auto-detects goal syntax (has parens) vs FTS search (plain text).
- LSP two-pass validation: pass 1 collects all names, pass 2 validates references. Declaration order doesn't matter.

## Key Insights
- link = materialized binary relation, fixed src/tgt schema, SQL predicates. query = virtual n-ary relation, variable unification, recursive CTEs. Collapsing them would force each to carry machinery it doesn't use.
- Recursion requires heterogeneous link chains (two+ link kinds per step) because links connect different match kinds. same_file predicate enables within-file hops that make chains possible.
- Branch/repo unification is the same problem as segment captures in globs. `branch(release/$V)` filters by glob and captures the match. One code path handles filter-only, capture-only, and filter+capture.
- The user's real use case: crawl from helm chart -> git tag -> lockfile -> internal deps -> lib repos. 4-5 hops, each a different link kind. Query rules chain them declaratively.
- User questioning whether sprefa is just codeql/semgrep. Key differentiator: reactive relational index (extract + link + watch + rewrite) vs offline pattern search.

## Open Questions
- Should prefiltering be a RuleSet optimization pass or a language-level feature?
- How does cross-rev linking work? Same file at different tags = different file_ids. Need identity across revisions.
- When to materialize query results vs keep them as virtual CTEs?
- How to surface "latest branch with X" -- post-query sort by commit date, or language-level aggregation?
