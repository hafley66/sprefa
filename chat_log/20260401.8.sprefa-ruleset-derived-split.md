# Session: sprefa-ruleset-derived-split

## Goal
Split RuleSet into extraction-only rules and derived rules (links + queries), add is_check and negated fields as prep for query redesign.

## Current State
Committed as 265dc5e. RuleSet now holds only extraction rules. DerivedRules holds link_rules and query_rules. QueryDef has is_check: bool, QueryAtom has negated: bool (both default false, no behavioral change yet). All 325+ non-watch tests pass.

## Problem/Context
Continuing the query redesign plan from the group_id session. The user wanted the extraction/derived split to make data flow direction explicit at the type level: RuleSet feeds the write path (refs/strings/matches), DerivedRules feeds the read path (match_links, SQL CTEs). This was driven by the user's discomfort with link and query rules living alongside extraction rules in the same struct.

## Solution/Approach
- New DerivedRules struct in rules/types.rs
- RuleSet slimmed to just { schema, rules }
- lower_program returns (RuleSet, DerivedRules) tuple
- CLI uses a RuleFile compat struct for JSON/YAML backward compat (those formats still carry link_rules/query_rules at top level, deserialized then split)
- is_check and negated added with serde skip_serializing_if = is_false
