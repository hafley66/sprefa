# Session: sprefa-new-syntax-cross-refs

## Goal
Implement BOARD.md #2: new .sprf syntax `rule(name) { body };` with headless captures, cross-rule references, and scan annotations. Kill the old `rule name($CAPS) > body;` syntax that was poisoning context windows.

## Current State
Committed as cafb2b2. Parser, AST, lowering, LSP, tests, sprefa-rules.sprf, and README all updated. 215 tests pass, 5 discovery_e2e tests fail on pre-existing Store migration issue (query old matches/match_labels tables that SqliteStore::flush_batch no longer populates).

## What Changed
- `RuleDecl` lost `captures` field -- captures inferred from `$VAR`s in body during lowering
- `CrossRef` + `CrossRefBinding` AST nodes added for `rulename(col: $VAR)` syntax
- `RuleBody::Ref` variant added for cross-rule references with optional child blocks
- `Tag::Scan` added -- `scan(repo: $VAR, rev: $VAR)` marks captures as scan-driving without context steps
- `collect_scan_annotations` in lowering detects repo()/rev() tags AND scan() declarations
- Parser rewrites: `strip_rule_keyword`, `try_parse_cross_ref`, `parse_cross_ref_bindings`
- `parse_rule_body_list` is now unused (body parsing goes through `parse_block_contents` from the `{` in `rule(name) {`)

## Use Cases Discussed
1. Helmfile gotmpl -> repo+rev extraction -> package.json dep crawl -> lock file rev tracking
2. OpenAPI spec as source of truth -> frontend callsite tracking by op name or URL pattern
3. Cross-repo hardcoded string sync detection on a timer

All expressible with cross-rule refs: `rule(A) { extract }; rule(B) { A(col: $VAR) { deeper extraction } };`

## Key Insights
- `scan()` tag was invented this session to solve the annotation gap: old system put repo/rev annotations in the head, new headless syntax needs them somewhere. `scan(repo: $VAR, rev: $VAR)` is annotation-only, no context step emitted.
- Cross-rule refs serve dual purpose: dependency edge in extraction context, table scan in query context. Same syntax, two execution contexts.
- Discovery E2E tests fail because SqliteStore::flush_batch writes to per-rule tables, not old matches/match_labels. Pre-existing issue from Store trait migration.
- `parse_rule_body_list` became dead code because `rule(name) { ... }` uses `parse_block_contents` directly after consuming the `{`.
