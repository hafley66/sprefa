# Session: sprefa-stable-ids-match-links-rule-rename

## Goal
Engine walkthrough, stabilize ref/match IDs for future match_links table, rename emit->create_matches, add re: regex prefix support to all pattern fields.

## Current State
All three changes implemented, building, 291 tests passing. Two commits made (stable IDs, query crate removal). emit->create_matches and re: prefix changes are unstaged.

## Problem/Context
Loaded session 7 (sql cmd, cleanup, docs). Did full linear engine walkthrough (scan pipeline 11 steps, watch pipeline reactive topology). Identified link rules as critical path. Discovered that replace_file_refs in watcher.rs was deleting and reinserting both refs and matches on every re-extraction, making their IDs unstable and unusable as FK targets for a future match_links junction table.

User challenged the initial plan to use ref_links (linking refs) instead of match_links (linking matches), pointing out that the link is between semantic interpretations (import_name -> export_name), not byte positions. Then challenged why matches were deleted but not upserted when the same argument applied to both layers.

## Solution/Approach

### Stable IDs (committed: 35d56b3)
Changed replace_file_refs from delete-all+insert-all to upsert+prune for both refs and matches. Refs upsert via ON CONFLICT(file_id, string_id, span_start) DO UPDATE. Matches upsert via ON CONFLICT(ref_id, rule_name, kind) DO NOTHING. Stale rows pruned after upsert. Both layers now preserve IDs across re-extraction.

### match_links design (not yet implemented)
Decided on match_links junction table (source_match_id, target_match_id, link_kind TEXT). Links semantic matches, not physical refs. First link_kind will be "import_binding" (import_name -> export_name). Populate pass runs after resolve_import_targets.

### emit -> create_matches rename (unstaged)
EmitRef type renamed to MatchDef. Field emit renamed to create_matches on Rule and CompiledRule. emit_refs() renamed to create_refs(). serde(alias = "emit") for backwards compat. sprefa-rules.json updated. 3 snapshots auto-updated.

### re: prefix for regex patterns (unstaged)
All pattern fields (file, folder, key_match, repo, branch, tag) now support re: prefix for regex. Bare patterns remain glob. Applied across pipe_glob_matches, CompiledPipeGlob, CompiledFileSelector, CompiledGitSelector. Named capture groups via (?P<name>...) work in regex mode.

## Key Insights
- Ref IDs are stable on UNIQUE(file_id, string_id, span_start) -- survives re-extraction in both scan and watch paths
- Match IDs are stable on UNIQUE(ref_id, rule_name, kind) -- depends on ref ID stability
- Links belong at the match layer (semantic) not the ref layer (physical) because the link is between interpretations
- parent_key_string_id is intra-file sibling linking, target_file_id is cross-file path linking, match_links will be cross-file semantic linking
- The delete+reinsert pattern was unnecessary -- same upsert logic applies to both refs and matches
- serde(alias) provides backwards compat for JSON field renames without breaking existing rule files

## Open Questions
