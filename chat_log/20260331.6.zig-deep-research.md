## 1. Type System

### Primitive Types

| Zig | TypeScript | Rust | Notes |
|-----|-----------|------|-------|
| `u8`, `u16`, `u32`, `u64` | `number` | `u8`, `u16`, `u32`, `u64` | |
| `i8`, `i16`, `i32`, `i64` | `number` | `i8`, `i16`, `i32`, `i64` | |
| `usize` | N/A | `usize` | Pointer-width unsigned |
| `f16`, `f32`, `f64`, `f80`, `f128` | `number` | `f32`, `f64` | Zig has more float widths |
| `bool` | `boolean` | `bool` | |
| `void` | `void` | `()` | |
| `noreturn` | `never` | `!` | |
| `comptime_int` | N/A | N/A | Arbitrary-precision, compile-time only |
| `comptime_float` | N/A | N/A | |
| `type` | N/A | N/A | First-class types at comptime |

Arbitrary-width integers: `u3`, `i7`, `u24` -- any width is valid. Integer literals are `comptime_int` (arbitrary precision) until coerced:

```zig
const x = 42;           // comptime_int, not i32
const y: u8 = x;        // coerces to u8
const z: i64 = x;       // same literal, coerces to i64
const big = 1 << 200;   // fine at comptime
```

### Optional Types (`?T`)

| Zig | TypeScript | Rust |
|-----|-----------|------|
| `?T` | `T \| undefined` | `Option<T>` |
| `null` | `undefined` | `None` |
| `orelse` | `??` / `\|\|` | `.unwrap_or()` |
| `if (x) \|val\|` | `if (x !== undefined)` | `if let Some(val) = x` |
| `.?` | `x!` (assertion) | `.unwrap()` |

```zig
var maybe: ?i32 = 42;
maybe = null;

const val = maybe orelse 0;        // unwrap with default
if (maybe) |value| { use(value); } // pattern match
const forced = maybe.?;            // unwrap or panic
```

Key: `?*T` has zero overhead -- null uses the 0 address, same as Rust's niche optimization on `Option<&T>`.

### Error Sets and Error Unions

| Zig | Rust |
|-----|------|
| `error{ A, B, C }` | `enum MyError { A, B, C }` |
| `E!T` (error union) | `Result<T, E>` |
| `try expr` | `expr?` |
| `catch \|err\|` | `.map_err()` / `match Err(e)` |
| `anyerror` | `Box<dyn Error>` (roughly) |
| `errdefer` | N/A (Drop handles cleanup) |

```zig
fn openFile(path: []const u8) FileError!File {
    if (!exists(path)) return error.NotFound;
    // ...
}

fn readConfig() !Config {
    const file = try openFile("config.txt"); // propagate error (like Rust's ?)
    defer file.close();
    return parseConfig(try file.readAll());
}
```

`errdefer` runs cleanup only on error return path:

```zig
fn createResource(allocator: Allocator) !*Resource {
    const ptr = try allocator.create(Resource);
    errdefer allocator.destroy(ptr); // only runs if function returns error
    try ptr.init();
    return ptr;
}
```

### Enums and Tagged Unions

```zig
// Enum with methods
const Suit = enum {
    clubs, diamonds, hearts, spades,
    pub fn isRed(self: Suit) bool {
        return self == .hearts or self == .diamonds;
    }
};

// Tagged union (like Rust enum with data)
const Expr = union(enum) {
    literal: i64,
    identifier: []const u8,
    binary: struct { left: *Expr, op: u8, right: *Expr },
    none: void,
};

fn eval(expr: Expr) i64 {
    return switch (expr) {
        .literal => |val| val,
        .binary => |bin| applyOp(bin.op, eval(bin.left.*), eval(bin.right.*)),
        .identifier => |name| lookupVar(name),
        .none => 0,
    };
    // exhaustive switch enforced by compiler
}
```

### Type Coercion Rules

Implicit coercions that exist:
- `comptime_int`/`comptime_float` -> concrete types (if value fits)
- Integer widening only (`u8` -> `u16`, never narrowing)
- `*[N]T` -> `[]T` (array pointer to slice)
- `*T` -> `*const T`
- `T` -> `?T` and `T` -> `E!T`

Everything else requires explicit builtins: `@intCast`, `@floatCast`, `@ptrCast`, `@truncate`.

## 2. Pointers

| Type | Meaning | Rust Equivalent |
|------|---------|----------------|
| `*T` | Single-item mutable pointer | `&mut T` |
| `*const T` | Single-item const pointer | `&T` |
| `[*]T` | Many-item pointer (C-style, no bounds) | `*mut T` (raw) |
| `[]T` | Slice (pointer + length) | `&mut [T]` |
| `[]const T` | Const slice | `&[T]` |
| `?*T` | Nullable pointer (zero overhead) | `Option<&mut T>` |
| `[*:0]const u8` | Null-terminated pointer (C string) | `*const c_char` |

```zig
var x: i32 = 10;
var ptr: *i32 = &x;     // take address
ptr.* = 20;             // dereference (Rust: *ptr = 20)

var array = [_]i32{ 1, 2, 3, 4, 5 };
const slice: []i32 = array[1..4]; // elements 1,2,3
// slice.len == 3, bounds-checked in safe modes

// String literals are *const [N:0]u8, coerce to []const u8
const s: []const u8 = "hello";
```

Pointer arithmetic available on `[*]T` only (not `*T`):

```zig
var buf: [100]u8 = undefined;
var ptr: [*]u8 = &buf;
ptr += 10;       // advance 10 elements
const val = ptr[0]; // read at offset
```

Sentinel-terminated types: `[3:0]u8` has 3 elements + a trailing 0. `[:0]const u8` is a null-terminated slice (C string with known length).

## 3. Memory Allocation

### The Allocator Interface

Every stdlib function that needs heap memory takes `std.mem.Allocator`. No global allocator (unlike Rust's `#[global_allocator]`). This is a vtable-based interface:

```zig
// Zig: allocator is explicit at every call site
fn buildList(allocator: Allocator, items: []const Item) !ArrayList(Item) {
    var list = ArrayList(Item).init(allocator);
    for (items) |item| try list.append(item);
    return list;
}

// Rust: allocator is implicit via global
fn build_list(items: &[Item]) -> Vec<Item> {
    items.to_vec() // uses global allocator
}
```

### Allocator Types

| Allocator | Use Case | Analog |
|-----------|----------|--------|
| `page_allocator` | Backing allocator, OS pages | `mmap` directly |
| `GeneralPurposeAllocator` | Default "just works", leak/UAF detection in debug | Rust's default allocator + miri |
| `ArenaAllocator` | Bulk-free, request-scoped | `bumpalo` crate |
| `FixedBufferAllocator` | No heap, stack/static buffer | Stack allocation |
| `c_allocator` | C interop (`malloc`/`free`) | libc |

```zig
// Arena: allocate freely, free everything at once
var arena = std.heap.ArenaAllocator.init(std.heap.page_allocator);
defer arena.deinit(); // frees ALL allocations

const allocator = arena.allocator();
const a = try allocator.alloc(u8, 100);  // no individual free needed
const b = try allocator.alloc(u32, 50);
```

### `defer` for Cleanup (vs Rust's RAII)

Zig has no destructors. Cleanup is manual via `defer` (runs at scope exit) and `errdefer` (runs only on error exit):

```zig
fn process(allocator: Allocator) !void {
    const buf = try allocator.alloc(u8, 1024);
    defer allocator.free(buf);

    const file = try std.fs.cwd().openFile("data.bin", .{});
    defer file.close();

    try doWork(buf, file);
    // defers run here in LIFO order
}
```

Tradeoff: `defer` is more flexible than RAII (supports error-conditional cleanup via `errdefer`), but forgettable -- omitting a `defer` causes a leak. Rust's `Drop` makes forgetting impossible.

## 4. C Interop

### Importing C Headers

```zig
const c = @cImport({
    @cInclude("stdio.h");
    @cInclude("sqlite3.h");
});

pub fn main() void {
    _ = c.printf("Hello from C! %d\n", @as(c_int, 42));
}
```

`@cImport` runs Zig's built-in C-to-Zig translator at compile time. The C header IS the binding -- no bindgen step, no extern block declarations.

**Rust comparison**: Rust requires `bindgen` (external tool) or manual `extern "C"` blocks. Zig eliminates this entirely.

### Compiling C Sources

```zig
// build.zig
exe.addCSourceFiles(.{
    .files = &.{ "vendor/sqlite3.c", "vendor/miniz.c" },
    .flags = &.{ "-std=c99", "-O2" },
});
exe.linkLibC();
exe.linkSystemLibrary("sqlite3");
```

### Zig as C Compiler (`zig cc`)

```bash
zig cc -o hello hello.c                        # compile C
zig cc -target aarch64-linux-gnu hello.c       # cross-compile
CC="zig cc" make                               # drop-in replacement
```

Uber uses `zig cc` in production for cross-compiling their entire Go monorepo's C/C++ dependencies.

### Type Mapping

| C | Zig |
|---|-----|
| `int` / `unsigned int` | `c_int` / `c_uint` |
| `size_t` | `usize` |
| `char*` | `[*:0]u8` |
| `const char*` | `[*:0]const u8` |
| `void*` | `*anyopaque` |
| `NULL` | `null` (in `?*T`) |

## 5. Functions

### Basics

```zig
fn add(a: i32, b: i32) i32 {
    return a + b;
}

// Multiple return via anonymous struct
fn divmod(a: u32, b: u32) struct { quot: u32, rem: u32 } {
    return .{ .quot = a / b, .rem = a % b };
}
```

Parameters are always immutable. Mutation requires a pointer parameter (`*T`).

### Generic Functions

Two mechanisms, no angle brackets:

```zig
// anytype: duck-typed at monomorphization (like C++ templates)
fn lessThan(a: anytype, b: anytype) bool {
    return a < b;
}

// Explicit comptime type parameter (like Rust's turbofish)
fn max(comptime T: type, a: T, b: T) T {
    return if (a > b) a else b;
}
const m = max(u32, 10, 20);

// Functions returning types: the core generic container pattern
fn ArrayList(comptime T: type) type {
    return struct {
        items: []T,
        capacity: usize,
        pub fn append(self: *@This(), item: T) !void { ... }
    };
}
var list = ArrayList(u32).init(allocator);
```

| | Zig `anytype` | Rust `T: Trait` | TS `<T>` |
|-|---|---|---|
| Constraint | Duck typing at instantiation | Trait bounds at definition | `extends` at definition |
| Error location | Call site (C++ template-like) | Definition site | Definition site |
| IDE support in generic body | Limited | Full (traits inform completions) | Full |

### No Closures

Zig deliberately has no closures. Rationale: no hidden allocation, no hidden control flow, comptime function values must be comptime-known.

**The context pointer pattern (closure replacement):**

```zig
const Callback = struct {
    context: *anyopaque,
    func: *const fn (context: *anyopaque, value: i32) void,

    fn call(self: Callback, value: i32) void {
        self.func(self.context, value);
    }
};

const Counter = struct {
    count: u32 = 0,

    fn onEvent(ctx: *anyopaque, value: i32) void {
        const self: *Counter = @ptrCast(@alignCast(ctx));
        self.count += 1;
        _ = value;
    }

    fn asCallback(self: *Counter) Callback {
        return .{ .context = @ptrCast(self), .func = &onEvent };
    }
};
```

This is what TypeScript `(value) => { this.count++ }` and Rust `|value| { self.count += 1 }` (`FnMut`) decompose into manually.

### Function Pointers as Arguments

```zig
// comptime function pointer: monomorphized, inlinable
fn apply(x: i32, y: i32, comptime f: fn (i32, i32) i32) i32 {
    return f(x, y);
}

// Anonymous function pattern (nearest lambda equivalent)
std.mem.sort(i32, &items, {}, struct {
    fn cmp(_: void, a: i32, b: i32) bool { return a < b; }
}.cmp);
```

## 6. Structs

```zig
const Point = struct {
    x: f64,
    y: f64,
    z: f64 = 0.0, // default value

    const Self = @This();

    pub fn distance(self: Self, other: Self) f64 {
        const dx = self.x - other.x;
        const dy = self.y - other.y;
        return @sqrt(dx * dx + dy * dy);
    }

    pub fn translate(self: *Self, dx: f64, dy: f64) void { // mutation: *Self
        self.x += dx;
        self.y += dy;
    }

    pub fn origin() Self { // "static" method
        return .{ .x = 0, .y = 0 };
    }
};
```

- Methods are functions in the struct namespace
- `self: Self` for read-only, `self: *Self` for mutation
- No `class`, `impl`, or `trait` keywords
- Zig merges Rust's `struct` + `impl` into one block

### Special Struct Kinds

```zig
// Packed struct: bit-level layout, no padding
const Flags = packed struct {
    readable: bool,    // 1 bit
    writable: bool,    // 1 bit
    executable: bool,  // 1 bit
    _reserved: u5 = 0, // 5 bits = 8 bits total
};

// Extern struct: C ABI layout (like Rust's #[repr(C)])
const CPoint = extern struct { x: c_int, y: c_int };
```

## 7. OOP Patterns in Zig

No inheritance, no virtual methods, no abstract classes.

### The Interface Pattern (Manual Vtables)

Runtime polymorphism via `{ *anyopaque, *const VTable }` -- architecturally identical to Rust's `dyn Trait`:

```zig
const Writer = struct {
    context: *anyopaque,
    writeFn: *const fn (context: *anyopaque, bytes: []const u8) !usize,

    pub fn write(self: Writer, bytes: []const u8) !usize {
        return self.writeFn(self.context, bytes);
    }
};

// Implementation
const FileWriter = struct {
    fd: i32,

    fn writeImpl(ctx: *anyopaque, bytes: []const u8) !usize {
        const self: *FileWriter = @ptrCast(@alignCast(ctx));
        return std.os.write(self.fd, bytes);
    }

    fn writer(self: *FileWriter) Writer {
        return .{ .context = @ptrCast(self), .writeFn = &writeImpl };
    }
};
```

| Mechanism | Zig | Rust | TypeScript |
|---|---|---|---|
| Contract | Struct with fn pointer fields | `trait` keyword | `interface` keyword |
| Static dispatch | `comptime` / `anytype` | `impl Trait` | N/A |
| Dynamic dispatch | Manual fat pointer | `dyn Trait` (auto vtable) | Structural typing |
| Compiler verifies contract | At use site only | At definition site | At assignment |

### Composition Over Inheritance

```zig
const Dog = struct {
    animal: Animal,      // embedded, not inherited
    breed: []const u8,

    pub fn fetch(self: Dog) void {
        std.debug.print("{s} fetches\n", .{self.animal.name});
    }
};

d.animal.speak(); // explicit delegation, no implicit method resolution
```

## 8. FP Patterns in Zig

### Higher-Order Functions

```zig
// std.mem.sort: comptime function pointer, zero-cost dispatch
std.mem.sort(i32, &items, {}, struct {
    fn cmp(_: void, a: i32, b: i32) bool { return a < b; }
}.cmp);
```

The `context` parameter is the explicit environment that a closure would capture implicitly.

### No Map/Filter/Reduce

Deliberate omission. `map` produces a new collection -- where does it allocate? Zig makes this explicit:

```zig
// TypeScript: items.filter(x => x > 3).map(x => x * 2)
// Rust:       items.iter().filter(|x| *x > 3).map(|x| x * 2).collect()
// Zig:
var result = std.ArrayList(i32).init(allocator);
for (items) |x| {
    if (x > 3) try result.append(x * 2);
}
```

### Observable Pattern (RxJS Analog)

```zig
pub fn Observable(comptime T: type) type {
    return struct {
        const Self = @This();

        pub const Subscription = struct {
            context: *anyopaque,
            callback: *const fn (context: *anyopaque, value: T) void,
            next: ?*Subscription = null,
        };

        head: ?*Subscription = null,
        allocator: std.mem.Allocator,

        pub fn init(allocator: std.mem.Allocator) Self {
            return .{ .allocator = allocator };
        }

        pub fn subscribe(self: *Self, sub: *Subscription) void {
            sub.next = self.head;
            self.head = sub;
        }

        pub fn emit(self: *Self, value: T) void {
            var current = self.head;
            while (current) |sub| {
                sub.callback(sub.context, value);
                current = sub.next;
            }
        }
    };
}

// Subscriber implementation
const Logger = struct {
    prefix: []const u8,
    count: u32 = 0,

    fn onValue(ctx: *anyopaque, value: i32) void {
        const self: *Logger = @ptrCast(@alignCast(ctx));
        self.count += 1;
    }

    fn subscription(self: *Logger) Observable(i32).Subscription {
        return .{ .context = @ptrCast(self), .callback = &onValue };
    }
};
```

| RxJS | Zig |
|---|---|
| `new Subject<number>()` | `Observable(i32).init(allocator)` |
| `.subscribe(observer)` | `stream.subscribe(&sub)` |
| `.next(value)` | `stream.emit(value)` |
| `{ next: (v) => ... }` | `Subscription{ .context = ptr, .callback = fn }` |
| Closure captures `this` | Explicit `@ptrCast(@alignCast(ctx))` |

RxJS operators (`map`, `debounce`, `switchMap`) would each be a struct holding a downstream `Subscription` and transforming values in its callback. Doable, verbose.

## 9. Comptime and Metaprogramming

### The Core Idea

`comptime` is a single keyword that replaces generics, macros, const generics, and type-level computation. Regular Zig code runs at compile time.

```zig
// Generics ARE comptime
fn List(comptime T: type) type {
    return struct {
        items: []T,
        len: usize,
        fn append(self: *@This(), item: T) void { ... }
    };
}
const IntList = List(i32); // compiler calls function, uses returned struct
```

### Type Reflection

`@typeInfo(T)` returns a tagged union describing T's structure. `@Type(info)` does the inverse.

```zig
// Iterate struct fields at compile time
fn dumpFields(comptime T: type) void {
    const info = @typeInfo(T);
    inline for (info.@"struct".fields) |field| {
        @compileLog(field.name, field.type);
    }
}

// Construct types programmatically
fn MakeVec(comptime n: usize) type {
    var fields: [n]std.builtin.Type.StructField = undefined;
    const names = [_][]const u8{ "x", "y", "z", "w" };
    for (0..n) |i| {
        fields[i] = .{
            .name = names[i], .type = f32,
            .default_value_ptr = null, .is_comptime = false,
            .alignment = @alignOf(f32),
        };
    }
    return @Type(.{ .@"struct" = .{
        .layout = .auto, .fields = &fields, .decls = &.{}, .is_tuple = false,
    }});
}
const Vec3 = MakeVec(3); // struct { x: f32, y: f32, z: f32 }
```

This is TypeScript's `keyof`/mapped types but at the value level using ordinary loops.

### Comptime JSON Serializer (replaces Rust's `#[derive(Serialize)]`)

```zig
fn jsonStringify(value: anytype, writer: anytype) !void {
    const T = @TypeOf(value);
    const info = @typeInfo(T);
    switch (info) {
        .@"struct" => |s| {
            try writer.writeAll("{");
            inline for (s.fields, 0..) |field, i| {
                if (i > 0) try writer.writeAll(",");
                try writer.print("\"{s}\":", .{field.name});
                try jsonStringify(@field(value, field.name), writer);
            }
            try writer.writeAll("}");
        },
        .int, .comptime_int => try writer.print("{d}", .{value}),
        .bool => try writer.writeAll(if (value) "true" else "false"),
        .pointer => |p| {
            if (p.size == .slice and p.child == u8)
                try writer.print("\"{s}\"", .{value});
        },
        .optional => {
            if (value) |v| try jsonStringify(v, writer)
            else try writer.writeAll("null");
        },
        else => try writer.writeAll("null"),
    }
}
```

| Aspect | Zig comptime | Rust derive macros |
|---|---|---|
| Code location | Same file, same language | Separate proc-macro crate |
| Debugging | Readable Zig code | TokenStream manipulation |
| Transparency | Fully visible | Requires `cargo expand` |

### Comptime Route Matching (FastAPI analog)

```zig
fn Router(comptime routes: []const Route) type {
    return struct {
        fn dispatch(path: []const u8, ctx: *Context) !void {
            inline for (routes) |route| {
                if (matchPath(route.pattern, path)) |params| {
                    return route.handler(ctx, params);
                }
            }
            return error.NotFound;
        }
    };
}

const app = Router(&.{
    .{ .pattern = "/users/:id",       .handler = getUser },
    .{ .pattern = "/users/:id/posts", .handler = getUserPosts },
});
```

### Comptime Lookup Tables

```zig
const CRC32_TABLE = comptime blk: {
    var table: [256]u32 = undefined;
    for (0..256) |i| {
        var crc: u32 = @intCast(i);
        for (0..8) |_| {
            crc = if (crc & 1 == 1) (crc >> 1) ^ 0xEDB88320 else crc >> 1;
        }
        table[i] = crc;
    }
    break :blk table;
};
```

### Limits of Comptime

| Cannot do at comptime | Rationale |
|---|---|
| I/O (filesystem, network) | Hermetic, reproducible builds |
| Call extern/C functions | No FFI at compile time |
| Runtime allocation | Comptime uses compiler-managed memory |
| Assembly | `asm` blocks forbidden |
| Access runtime values | Only comptime-known data visible |

Branch quota: default 1000 backward branches. Override with `@setEvalBranchQuota(10000)`.

### Comparison

| Capability | Zig comptime | Rust const fn + proc macros |
|---|---|---|
| Arbitrary types as params | Yes (`comptime T: type`) | No (only primitives as const generics) |
| Type reflection | `@typeInfo` | Requires proc macros |
| Return new types | Yes | Cannot construct types in const fn |
| String processing | Yes, arbitrary | No string const generics |
| Replace macros | Mostly yes | No, proc macros still needed |

## 10. Async, Concurrency, Parallelism

### Current State of Async

Zig had `async`/`await` through 0.10. **Removed in 0.11.** The replacement, `std.Io`, is landing in 0.15-0.16:

- Every I/O function accepts an `Io` parameter (like `Allocator`)
- Caller picks the concurrency backend: blocking threads, io_uring, kqueue, IOCP
- No function coloring -- code reads like synchronous Zig
- No new language keywords

| Language | Async model |
|----------|-------------|
| Rust | Colored functions (`async fn` -> `Future`), requires executor (tokio) |
| TypeScript | Single-threaded event loop, `async`/`await`+`Promise` |
| Go | Goroutines, M:N threading hidden from programmer |
| Zig (planned) | No function coloring. `Io` interface injected like allocator. |

### What Developers Use NOW (0.13/0.14)

**std.Thread:**
```zig
fn worker(id: usize) void {
    std.debug.print("thread {d}\n", .{id});
}

pub fn main() !void {
    var threads: [4]std.Thread = undefined;
    for (&threads, 0..) |*t, i| {
        t.* = try std.Thread.spawn(.{}, worker, .{i});
    }
    for (&threads) |t| t.join();
}
```

**std.Thread.Pool:**
```zig
var pool: std.Thread.Pool = undefined;
try pool.init(.{ .allocator = allocator, .n_jobs = 4 });
defer pool.deinit();

var wg: std.Thread.WaitGroup = .{};
for (0..10) |i| {
    pool.spawnWg(&wg, struct {
        fn run(id: usize) void { std.debug.print("task {d}\n", .{id}); }
    }.run, .{i});
}
wg.wait();
```

Synchronization primitives: `Mutex`, `RwLock`, `Condition`, `Semaphore`, `Futex`, `WaitGroup`.

### io_uring

Zig's stdlib includes io_uring bindings (contributed by TigerBeetle team):

```zig
var ring = try linux.IO_Uring.init(256, 0);
defer ring.deinit();

var sqe = try ring.get_sqe();
sqe.prep_read(fd, buffer, offset);
_ = try ring.submit_and_wait(1);
```

**libxev** (Mitchell Hashimoto): cross-platform event loop wrapping io_uring/kqueue/WASI poll. Used in Ghostty.

### SIMD via @Vector

```zig
const Vec8f = @Vector(8, f32);

pub fn addVectors(a: Vec8f, b: Vec8f) Vec8f {
    return a + b;  // single SIMD instruction
}

pub fn dotProduct(a: Vec8f, b: Vec8f) f32 {
    return @reduce(.Add, a * b);
}
```

`@Vector` is stable and first-class. Rust's `std::simd` is still nightly-only.

### URL/HTTP Parameter Handling

`std.Uri` parses URIs but does not parse query key-value pairs:

```zig
const uri = try std.Uri.parse("https://example.com/path?name=foo&age=42");
// uri.query == "name=foo&age=42" (raw string)

// Manual parsing:
var iter = std.mem.splitScalar(u8, uri.query, '&');
while (iter.next()) |pair| {
    if (std.mem.indexOfScalar(u8, pair, '=')) |eq| {
        const key = pair[0..eq];
        const value = pair[eq + 1 ..];
    }
}
```

http.zig provides `req.query().get("key")` with automatic percent-decoding.

| | Zig (std) | Zig (http.zig) | FastAPI | Express |
|-|---|---|---|---|
| Query parsing | Manual split | `.get("key")` | Automatic from type hints | `req.query.key` |
| Type coercion | Manual | Manual | Automatic (Pydantic) | Manual |
| Validation | Manual | Manual | Declarative | Manual |

### Build System (`build.zig`)

Written in Zig. Supports dependency graphs, parallel execution, C/C++ compilation, custom steps.

```zig
pub fn build(b: *std.Build) void {

    const optimize = b.standardOptimizeOption(.{});
    const exe = b.addExecutable(.{
        .name = "myapp",
        .root_source_file = b.path("src/main.zig"),
        .target = target, .optimize = optimize,
    });
    b.installArtifact(exe);
}
```

### Test Runner

Language-level feature. Tests colocated with source:

```zig
test "basic addition" {

// Run: zig test src/myfile.zig
```

`std.testing.allocator` detects memory leaks in tests.

### Cross-Compilation

~45MB toolchain bundles musl/glibc/Windows/macOS headers for ~40 targets:

```bash

zig build -Dtarget=x86_64-windows        # Windows from macOS
zig build -Dtarget=aarch64-linux-musl     # ARM Linux
zig build -Dtarget=wasm32-wasi            # WASI
```

No separate SDK downloads needed. This is a genuine killer feature.

## 13. Performance: Zig vs Rust vs C

### Why Zig Matches C

- Same LLVM backend as Clang
- No hidden control flow (no exceptions, no operator overloading, no hidden allocations)
- No runtime (no GC, no async runtime, no hidden thread)
- `ReleaseFast` mode produces equivalent code to C `-O2`/`-O3`

### Build Modes

| Mode | Optimizations | Safety Checks | Use |
|------|:---:|:---:|-----|
| Debug | None | All | Development |
| ReleaseSafe | Yes | Bounds, overflow, null | Production default |
| ReleaseFast | Full | None | Maximum perf |
| ReleaseSmall | Size-optimized | None | Embedded |

### Empirical Benchmarks

**General microbenchmarks**: Zig and Rust trade wins. Neither dominates. Both are within noise of C on compute-bound work.

**Bytecode interpreter** (zackradisic/rust-vs-zig): Zig 1.56-1.76x faster than Rust. Analysis: unsafe Rust's UB rules (pointer provenance, aliasing) constrain optimizations in pointer-heavy code that Zig's simpler model allows.

**HTTP throughput**: rust-hyper ~1.01x faster than zig-http-client. Essentially a tie.

**AWS benchmark (2026)**: Zig slightly higher throughput. Rust lower latency.

### Production Case Studies

| Project | Zig% | Performance Claim |
|---------|------|---|
| TigerBeetle (financial DB) | 100% | 1M transactions/s. 97K/s on Raspberry Pi 4 |
| Bun (JS runtime) | ~61% | 2.8x Node.js HTTP throughput. 4x faster startup |
| Ghostty (terminal) | 100% | SIMD-optimized parser, GPU rendering |
| Uber | zig cc | Cross-compilation toolchain for Go monorepo |

### Compilation Speed

| Metric | Before | After | Improvement |
|--------|--------|-------|:-----------:|
| Zig compiler self-build | 75s | 20s | 73% |
| Hello world | 22.8s | 275ms | 99% |
| Ghostty full build | 41s | 32s | 22% |

Rust+GTK hello world: 2.8 GB RAM to compile (~10x Zig equivalent). Zig is actively removing LLVM from the debug build path.

### When Zig Excels

- **C interop**: zero-cost, zero friction. Zig's C interop is built into the compiler
- **Cross-compilation**: out-of-the-box to ~40 targets, no SDK downloads
- **Embedded**: no runtime, predictable memory, static allocation "a breeze"
- **Compile time matters**: sub-second rebuilds on the trajectory
- **Explicit memory control**: "not using dynamic allocation is hard mode in Rust, but a breeze in Zig" (TigerBeetle)

### When Rust Wins

- **Memory safety at scale**: "multiple memory safety bugs per week" in Zig projects vs one bug per 140K lines of Rust
- **Fearless concurrency**: ownership prevents data races at compile time, Zig relies on discipline
- **Ecosystem**: orders of magnitude more libraries on crates.io
- **Abstraction optimization**: Rust iterator chains etc. optimize fully, ergonomic and zero-cost

### When C Wins

- **Existing codebases**: no FFI boundary to cross
- **Widest platform support**: GCC/Clang/MSVC + dozens of embedded compilers
- **Most mature tooling**: decades of debuggers, profilers, sanitizers

### Summary

```
Performance ceiling:  Zig = Rust = C  (same LLVM backend, same machine code)
Memory safety:        Rust >> Zig > C  (compile-time vs runtime vs nothing)
Compilation speed:    C > Zig >> Rust  (Zig actively closing gap with C)
C interop friction:   C < Zig << Rust  (Zig's @cImport vs Rust's bindgen)
Ecosystem breadth:    Rust >> C > Zig  (crates.io vs system libs vs early)
Cross-compilation:    Zig > Rust > C   (built-in vs toolchain install vs pain)
```
