# Session: sprefa-match-links-generic-link-rules

## Goal
Implement match_links table + generic link rule system for cross-file/cross-repo semantic linking between matches.

## Current State
All implemented, building, 297 tests passing. Two logical changes ready to commit:
1. match_links table + initial hardcoded resolve_match_links (then refactored to generic)
2. Generic link rule system with raw SQL WHERE injection

## Problem/Context
Loaded session from stable-ids work. match_links was designed but not implemented. Started with hardcoded import_binding linking, then user pushed toward a generic system where link rules are user-programmable SQL WHERE fragments in sprefa-rules.json. Key design evolution:
- Started with bespoke resolve passes per domain (resolve_image_refs, resolve_dep_versions)
- User pushed back: "those are bespoke"
- Moved to configurable link rules with match strategy enum (exact/norm/scoped)
- User pushed back: "how is that programmable by any lang/runtime"
- Arrived at raw SQL injection into a documented skeleton -- full expressiveness for prototyping

Also discussed broader vision: sprefa as cross-codebase causal linking engine ("ast-grep zoomed out"), not just rename refactoring. Updated project_vision.md memory.

## Solution/Approach

### match_links migration
CREATE TABLE IF NOT EXISTS match_links (source_match_id, target_match_id, link_kind) with UNIQUE constraint and indexes. Additive-only, no FK cascades from existing tables.

### LinkRule type (types.rs)
Two fields: `kind` (identifier + link_kind column value) and `sql` (raw WHERE fragment). Heavily documented skeleton on RuleSet::link_rules showing all available column aliases.

### Generic resolve_match_links (match_links.rs)
Takes &[LinkRule], loops, injects each sql into the skeleton. Single INSERT OR IGNORE per rule. Idempotent via NOT EXISTS guard. Logs per-rule counts. Errors include the SQL fragment for debugging.

### Wiring

- CLI build_scanner loads RuleSet first, extracts link_rules, passes both
- sprefa-rules.json has first link rule: import_binding
- ScanResult + ScanResultItem + CLI output carry links_created count

## Key Insights
- repo_packages table is scaffolding from before match/link system -- generic link rules subsume it (dep_name -> package_name by norm)
- "repos are strings too" -- a repo's manifest declares its name as a ref+match, so match_links connects consumers to declarations same as import->export
- Raw SQL injection is the right prototype tradeoff for a local toolchain -- bounded by the documented skeleton, unbounded in predicate expressiveness
- LinkRule.name was redundant with LinkRule.kind -- user caught the near-duplicate, dropped name

## Open Questions
- Watch integration: should link rules re-run after replace_file_refs on content changes?
- Link rule DSL design: what bounded predicate language replaces raw SQL?
- Cross-repo linking: current skeleton scopes source to one repo (src_rp.name = ?), target is unconstrained -- is that the right default?
