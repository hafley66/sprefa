# Session: sprf-lang-design-destructuring

## Goal
Design the .sprf language -- CSS-style selector chains with `>` combinator and a JSON destructuring mini-lang for structured data matching.

## Current State
Plan file written and user-edited at ~/.claude/plans/drifting-stargazing-jellyfish.md. Design is solidifying but NOT ready for code yet. Several open questions remain.

## Key Design Decisions (settled)
- `>` combinator between pattern slots (no keywords like from/where/select)
- `<<...>>` delimiters for embedded patterns (ast-grep, regex, destructuring)
- All captures must be SCREAMING_CASE: $KEY, $NAME, $VAL
- Every bare string is assumed glob until proven otherwise
- Quotes only escape spaces
- Selector chain order: repo > branch > fs > content > output
- Destructuring mini-lang operates on serde_json::Value (JSON/YAML/TOML)
- Partial object matching (unlisted keys ignored, unlike ast-grep)
- Ancestor captures carry forward through array iteration
- walk.rs stays untouched, coexists for JSON-loaded rules
- ast-grep stays for source code matching (ast engine)
- Regex stays for text matching (re engine)
- jq removed for now
- Link rules / query subcommand punted to future sessions

## Key Design Decisions (from previous session, still valid)
- File extension: .sprf
- Datalog is the compilation target / backend semantics, NOT the surface syntax
- CSS selector philosophy: compose matchers, not write queries

## Open Design Questions
- "literal" in value position of destructuring -- how does this interact with glob-by-default rule?
- Glob capture syntax `{pattern:-$VAR}` -- exact syntax TBD, bespoke
- How repo/branch context slots work in practice (less useful to capture since they come from matched path)
- Wiring destructuring Pattern into Rule type -- add field to Rule? Or separate path?

## What ast-grep CAN do for JSON/YAML (tested this session)
- `{"version": "$X"}` matches single-key objects (exact structural match)
- pair selector with context: `{"name": "$NAME"}` selector: pair -- captures key-value pairs
- Cross-sibling with inside/has/stopBy -- works but verbose YAML rules
- CANNOT do partial object match (extra keys = no match)
- CANNOT do TOML

## What the destructuring engine does better
- Partial object match: { name: $N } matches objects with extra keys
- Array iteration with ancestor carry-forward: { name: $N, items: [...{v: $X}] }
- Glob on keys: { dep_*: $VAL }
- Works on JSON/YAML/TOML uniformly (all parse to serde_json::Value)

## Key Insights
- ast-grep is exact structural matching -- great for source code, bad for "give me this key from any object"
- The destructuring mini-lang is essentially a tree pattern matcher with partial matching semantics
- Pattern AST doubles as the match spec -- no intermediate representation needed
- serde_json::Value is the universal tree representation for all structured formats
