# Session: sprefa-incremental-scan-diff-ghcache

## Goal
Implement incremental scan_diff for ghcache-triggered branch updates. When ghcache pulls main and reports a change, sprefa should use git2 to diff old_sha..new_sha and only re-extract changed files, not walk the entire tree.

## Current State
All implemented, building, 297 tests passing. Ready to commit.

## Problem/Context
Loaded from match_links session. User wants ghcache integration to be incremental -- staging dirs managed by ghcache should not use fs watchers, and rescans after pull should only touch changed files. The existing ghcache subscriber did a full scan_repo() on every checkout event, walking all files even when only 3 changed.

Key design points from user:
- Staging dirs don't checkout different branches, they pull and stay on main
- Use git2 to figure out what changed between old and new HEAD
- No scan should wipe anything -- additive/incremental SQL only
- Pure renames (same content, different path) should update the path on the existing file row, preserving file_id/refs/matches without re-extraction

## Solution/Approach

### diff_files() in crates/index/src/files.rs
- DiffResult struct with changed (abs paths), deleted (rel paths), renamed (old_rel, new_rel pairs), new_sha
- git2 diff_tree_to_tree between old_sha and HEAD
- Renamed deltas: if old/new OIDs match, it's a pure rename (no re-extraction needed). If OIDs differ, treat as delete + add.

### extract_files() in crates/index/src/extract.rs
- Refactored inner par_iter block into extract_from_list()
- New public extract_files() takes explicit path list instead of walking tree
- extract() now calls extract_from_list() internally (no behavior change)

### delete_branch_files_by_paths() in crates/cache/src/flush.rs
- Chunked DELETE from branch_files for removed paths
- Files/refs/matches rows stay (TODO: soft-delete/diff table for diagnostic holes later)

### rename_file_paths() in crates/cache/src/flush.rs
- UPDATE files SET path, stem, ext for pure renames
- Preserves file_id, refs, matches -- no re-extraction

### scan_diff() on Scanner in crates/scan/src/scanner.rs
- Takes old_sha, diffs via git2, handles deletes/renames/changed
- Extracts only changed files, flushes additively
- Returns ScanResult with new_git_hash for caller to persist

### ScanResult changes
- Added files_deleted, files_renamed, new_git_hash fields
- scan_repo() now populates new_git_hash by reading HEAD after flush

### ghcache subscriber wiring in crates/cli/src/main.rs
- Looks up prev sha via get_repo_branch_hash()
- Calls scan_diff when prev sha exists, falls back to scan_repo on error
- Persists new_git_hash via upsert_repo_branch after success
- Initial daemon scan also records git_hash for future incrementals

## Key Insights
- repo_branches.git_hash and get_repo_branch_hash/upsert_repo_branch already existed but weren't populated -- just needed wiring
- git2 Delta::Renamed with matching OIDs = pure rename, no extraction needed, just path update
- flush() is already additive (INSERT OR IGNORE) for committed branches, so passing a subset of changed files works perfectly
- sprefa_schema is only a dev-dep of crates/scan, so SHA persistence logic stays in the CLI caller
