# Session: sprefa-rule-engine

## Goal
Build the declarative rule engine for sprefa -- JSON rules that define how strings in structured files reference things in other repos.

## Current State
Rule engine core is complete and committed (e5c83fa). 36 tests passing (9 unit + 27 integration).

## Solution/Approach
DOM mental model: entire indexed space is a tree (root > repos > files > parsed tree nodes). Rules are CSS-style selectors with three dimensions:
1. Git context (repo/branch/tag globs)
2. File path (glob)
3. Structural position (step chain: Key, KeyMatch, Any, Depth*, ParentKey, ArrayItem, Leaf, Object)

Key design decisions:
- Steps capture values by name as they match (like regex named groups but for tree walking)
- Object step captures sibling values from one node (repository + tag from same image block)
- Value regex splits captures (scoped packages: @scope/name@version)
- Action emit array turns captures into grouped refs with parent_key linkage
- JSON Schema generated from Rust types via schemars for IDE intellisense

## Key Insights
- The "Any" step is the only non-trivial part of walk -- it forks into "consume and keep trying" vs "stop and advance"
- parent_key in the DB schema already handles grouped refs (name + version as linked refs)
- Dedup at DB insert handles duplicate captures from fan-out walks
- Value regex source field lets you pick which capture to split
- The greedy regex `(.+)@([^@]+)$` handles both scoped and unscoped npm packages
