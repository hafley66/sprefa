# Session: sprefa-scan-order-fix-datalog-discussion

## Goal
Fix scan-order dependency bug in cross-repo link resolution, then discuss the datalog-like semantics of the system.

## Current State
Fix is implemented and compiles, all tests pass (186 non-flaky). Not yet committed.

## Problem/Context
When scanning multiple repos, resolve_match_links runs per-repo during scan. If repo A scans before repo B, links from A to B's matches won't resolve because B isn't indexed yet. The INSERT OR IGNORE makes it idempotent, so a second pass catches stragglers.

## Solution/Approach
1. Added `Scanner::resolve_links(repo_name)` method -- thin wrapper around `sprefa_cache::resolve_match_links`
2. In `cmd_scan`, after the scan loop, when multiple repos were scanned and link rules exist, runs a second pass calling `scanner.resolve_links()` for each repo
3. Changed `for repo in repos` to `for repo in &repos` so repos is available for second pass
4. Only prints output if new links were actually created in second pass

## Key Insights
- The match_links table is a base relation; SQLite recursive CTEs and NOT EXISTS already provide datalog-like transitive closure, negation, and disjunction at query time
- No need to add recursion/negation/OR to the predicate DSL -- the rule engine produces ground facts, SQLite is the query evaluator
- The ergonomic gap is a query CLI/API for canned patterns (blast radius, orphans, dep tree) -- user has a prototype of this at work
- Link rules support both predicate DSL (guardrails) and raw SQL WHERE fragments (escape hatch)
- ghcache scan path is event-driven so second pass doesn't apply there; it self-heals on next scan of the earlier repo
- Live rename tests in sprefa_watch are pre-existing flaky (timeout), unrelated to this change
