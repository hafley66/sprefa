# Session: sprefa-sprf-language-design

## Goal
Design the .sprf query/rule language from scratch. Evolved from datalog-esque syntax to a unified tree selector language with pattern matching and captures at every level.

## Current State
Language design is converging. No code written yet. Plan file exists but needs update to reflect the tree-selector direction (diverged from pure datalog during this session). Six datalog skill files written for background knowledge.

## Problem/Context
sprefa-rules.json (YAML/JSON config) is too verbose and splits extraction/linking into different syntax shapes. User wants one language that unifies everything. Started with datalog, evolved through CSS selectors, landed on a tree path selector with captures.

## Solution/Approach

### The .sprf Language (as designed this session)

**Core insight**: everything is a tree. Repo tree → git tree → filesystem tree → content tree. One selector walks all levels. Captures ($NAME) punch holes at any level.

**Three statement types**:
```
from <glob> where <pattern> select <kinds> .     ← extraction (BigQuery pipe style)
link <label> :- <body> .                          ← linking (datalog)
query <name> :- <body> .                          ← query (datalog)
```

BUT: the `where` clause evolved into a full tree selector with nested destructuring:

```
repo_pattern > branch_pattern > file_glob > content_selector
```

**Content selector (JSON/YAML/TOML)**:

Tree path with captures at every level:
```
json.*.{
  name: $NAME,
  (devDependencies|dependencies): { $DEP: $VER },
  scripts: { build: { cmd: $BUILD_CMD } },
  workspaces: [...$WS_GLOB],
}
```

**Key position patterns** (uniform at every level):
- `"literal"` -- exact string
- `(a|b|c)` -- alternation
- `/regex/flags` -- regex
- `*.glob.*` -- glob
- `$CAP` -- wildcard + capture key name
- `*` -- wildcard, no capture

**Value capture**: `key: $CAPTURE` binds the value at that key
**Key+value capture**: `{ $KEY: $VAL }` binds both sides during object iteration
**Array spread**: `[...$ITEMS]` iterates all array elements, captures each
**Nested destructure**: `{ key: { subkey: $VAL } }` -- braces = object level, fully explicit

**Embedded patterns** (for source code files):
- `<<...>>` delimiters with engine tag: `ast <<pattern>>`, `re <<pattern>>`
- Same `$CAPTURE` syntax inside patterns

**Pattern matchers are uniform across ALL tree levels**:
- Repo names: glob, regex, alternation
- Branch names: regex with flags
- File paths: glob with `**` descent, ast-grep `$$$` style captures
- JSON keys: glob, regex, alternation, captures
- JSON values: type-aware capture

**Linking**: two selectors share a variable name → join on that column
```
rule deps: ... select dep_name($DEP), dep_version($VER)
rule packages: ... select pkg_name($NAME)
link dep_to_package: deps.$DEP = packages.$NAME
```

## Key Design Decisions
- File extension: `.sprf` ("cause it sounds funny")
- `<<>>` for embedded pattern blocks (no collision with any embedded language)
- Commas delimit, not `>`
- `from` always comes first (BigQuery pipe style: from → where → select)
- No shorthand paths -- every nesting level is explicit with `{}`
- Spread `...` for arrays (JS/Rust convention), not `$$$` (ast-grep convention)
- Key positions accept any pattern type (literal, glob, regex, alternation, capture)
- Every match fans out -- regex matching 3 keys produces 3 binding sets
- Concrete matched key stored alongside captures in DB
- Colon separates key pattern from value capture: `$DEP: $VER`

## Key Insights
- Tree-sitter queries are the closest prior art (S-expression patterns with @captures over trees)
- No existing JSON query language captures both key and value with named holes
- JSONPath, JMESPath, jq are all extraction-oriented not pattern-matching-oriented
- The tree path IS the query, captures ARE the columns, links ARE joins on shared column values
- gron (flatten JSON to path=value) is conceptually the inverse of what .sprf does
- Type annotations are redundant with descent -- the tree structure IS the type
- walk.rs already handles most of the JSON/YAML/TOML tree walking, .sprf is a frontend for it + the destructure/capture extensions

## Open Questions
- How exactly do link rules reference captures from extraction rules?
- Does destructuring need a "rest" pattern for unmatched keys?
- How to handle TOML-specific structure (tables, inline tables)?
- Array matching: single element match `[$PATTERN]` vs spread `[...$ALL]` -- is there a "match first" or "match Nth"?
- How to assert leaf types (string vs number) without angle bracket annotations?
- Exact grammar for regex-in-key-position: `/pattern/flags` vs `re<<pattern>>`?
