# Session: repo-refs-file-predicates

## Goal
Make all string-valued metadata in the sprefa system linkable through the match_links system. Repo-level entities (repo name, git tags, branches) and file-level metadata (stem, ext, dir) should be referenceable by link rule predicates.

## Current State
IMPLEMENTED AND PASSING. All 47 cache tests green, 35 rules tests green, 2 e2e tests green, 108 watch unit tests green. Full workspace compiles clean. Not yet committed.

## Problem/Context
match_links requires both sides to be matches, which require refs, which require file_id NOT NULL. Repo-level metadata (repo name, git tags, branches) had no file anchor and couldn't participate in linking. File metadata (stem, ext, dir) existed on the files table but wasn't referenceable in link predicates.

## Solution/Approach
Two-pronged:

1. **repo_refs table** -- new table `repo_refs(id, string_id, repo_id, kind)` for repo-level entities. `matches.ref_id` becomes nullable, `matches.repo_ref_id` added with CHECK constraint ensuring exactly one is non-null. Link skeleton target side uses LEFT JOINs + COALESCE to resolve strings from either refs or repo_refs.

2. **File column predicates** -- new LinkPredicate variants (StemEq, ExtEq, DirEq) that compile to SQL comparing files.stem/ext/dir against the other side's string norm. Zero extra rows. Added files.dir column populated during flush.

Design discussions explored: sentinel meta-files (rejected by user), nullable file_id on refs (wider blast radius), and compound/relational refs (deferred as future direction).

## Key Insights
- `normalize()` is just `trim().to_lowercase()`, so `LOWER(files.stem)` is equivalent for file column predicates
- The CHECK constraint `(ref_id IS NULL) != (repo_ref_id IS NULL)` enforces the invariant at the DB level
- Source side of link skeleton stays INNER JOIN (always file-backed from extraction), only target side needs LEFT JOINs
- SQLite handles ON CONFLICT with partial unique indexes correctly
- DB is a droppable cache, so breaking schema changes (matches table rebuild) are acceptable
- File metadata refs (stem/ext/dir) as predicates against existing columns = zero row inflation vs materializing them as refs
