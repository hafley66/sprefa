## What sem is

Rust tool. Entity-level semantic diff for git repos. Uses tree-sitter for parsing, git2 for repo access, rayon for parallelism. Supports 21 languages + structured data formats (JSON, YAML, TOML, CSV, Markdown).

Instead of line-level diffs, extracts named code entities (functions, classes, structs, etc.) from before/after file states, matches them to produce added/modified/deleted/renamed/moved changes. Builds cross-file dependency graph for blast radius queries.

## Key data structures

```
SemanticEntity {
    id, file_path, entity_type, name,
    content, content_hash, structural_hash,
    start_line, end_line, parent_id, metadata
}

EntityGraph {
    forward: HashMap<EntityId, Vec<(EntityId, EdgeType)>>,   // what I depend on
    reverse: HashMap<EntityId, Vec<(EntityId, EdgeType)>>,   // who depends on me
}

EdgeType = Calls | TypeRef | Imports
```

Entity IDs are `filepath::type::name` (or `filepath::parent_id::name` for nested).

## CLI commands

| Command | Purpose |
|---|---|
| `diff` | Entity-level diff with rename detection. `--staged`, `--commit`, `--from/--to`, `--stdin`, `--format json` |
| `impact <entity>` | Transitive blast radius via BFS (capped 10k). `--deps`, `--dependents`, `--tests` modes |
| `blame <file>` | Entity-level git blame (who last modified each entity) |
| `log <entity>` | Entity evolution through git history, `--limit 50` |
| `entities <file>` | Dump all entities in a single file with types and line ranges |
| `context <entity>` | Token-budgeted context window for LLMs (default 8000 tokens). Packs entity + deps + dependents |
| `setup` / `unsetup` | Replace `git diff` globally with sem |

## How entity matching works (3-phase)

1. **Exact ID match**: same `file::type::name` in before/after
2. **Structural hash match**: same content under different name (rename/move detection). Structural hash strips comments/whitespace via xxhash
3. **Fuzzy similarity**: >80% token overlap for probable renames (hardcoded threshold)

## How dependency resolution works

Two-pass approach:
1. Extract all entities, build `name -> entity_id[]` symbol table
2. Tokenize each entity's content (split on non-alphanumeric), strip comments/strings, look up each identifier token in symbol table

Cross-file resolution relies on import table built by regex-parsing import statements per file. Maps `(file, imported_name)` to target entity ID. Without an import, same-file resolution only.

## Storage

SQLite cache at `.sem/cache.db`:
- `files` (path + mtime)
- `entities` (full entity data)
- `edges` (graph edges)

Cache invalidation is all-or-nothing: if any file's mtime differs or file count changes, entire cache discarded and rebuilt. No incremental per-file invalidation.

## Limitations

- Symbol resolution is name-based tokenization, not AST-aware. Splits content on non-alphanumeric chars, looks up words in global name table. No scope analysis, no qualified path resolution, no type inference
- Cross-file refs require explicit import statements that the regex parser recognizes. Implicit deps (re-exports, dynamic requires, module augmentation) invisible
- No incremental graph updates -- cache is all-or-nothing rebuild
- `entities` is single-file only; no repo-wide entity dump command
- No entity type filtering, no name pattern matching, no structured queries
- Does not track: variable mutations, control flow, data flow, type hierarchies (implements/extends), macro expansions, generated code, conditional compilation
- 80% fuzzy similarity threshold for rename detection is hardcoded

## Querying capabilities

Minimal. Entity lookup by exact name match against flat symbol table. Disambiguation by `--file` hint. `--file-exts` filtering on diff/impact/context. No glob/regex on entity names, no type filtering, no pattern queries.

This is where sprefa fills the gap. sem extracts; sprefa queries.

## Stack overlap with sprefa

| Component | sprefa | sem |
|---|---|---|
| Language | Rust | Rust |
| Parsing | tree-sitter (via ast-grep), oxc, syn | tree-sitter |
| Parallelism | rayon | rayon |
| Storage | SQLite | SQLite |
| Git access | git2 | git2 |
| Hashing | xxh3 | xxhash |
| Extraction unit | string refs at byte offsets | named entities at line ranges |
| Cross-file links | match_links (configurable predicates) | EntityGraph (name-based token lookup) |

## Integration paths

### Path 1: `cmd()` tag -- zero coupling

```sprf
rule(changed_entities) {
  cmd(sem diff --format json HEAD~1..HEAD) > json({ changes: [...{ entity_id: $ID, change_type: $TYPE }] })
};
```

sem stays external binary. sprefa parses JSON output. No library dependency. Available once `cmd()` tag lands.

### Path 2: Per-file extractor -- during scan

sprefa's extractors work per-file. sem's entity extraction also works per-file via tree-sitter. Register sem as another extractor alongside JS/Rust/rule-engine:

```
scan pipeline:
  git ls-files
    -> parallel rayon walk
    -> per-file extraction:
        JS/TS (oxc)        -> imports, exports, require
        Rust (syn)         -> use paths, declarations
        Rule engine        -> configurable tree walks
        ast-grep           -> structural patterns
        sem entities       -> functions, classes, structs, fields  <-- NEW
    -> bulk flush to SQLite
    -> resolve links (sprefa match_links + sem EntityGraph edges)
```

This gives entity-level granularity in the same scan pass. sem's entities become rows in per-rule tables. sem's dependency edges become link table rows.

### Path 3: Shared SQLite -- read sem's cache

Both use SQLite. sprefa could ATTACH sem's `.sem/cache.db` and query it directly:

```sql
ATTACH DATABASE '.sem/cache.db' AS sem;
SELECT * FROM sem.entities WHERE entity_type = 'function';
```

No extraction needed. But depends on sem's cache being populated and its schema staying stable.

### Path 4: EntityGraph -> TreeNode adapter

sem's entity containment tree (file -> class -> method -> field) maps to `TreeNode`:

```
TreeNode::Map([
    ("file_path", Leaf("src/auth.ts")),
    ("entities", List([
        Map([
            ("type", Leaf("class")),
            ("name", Leaf("AuthService")),
            ("children", List([
                Map([("type", Leaf("method")), ("name", Leaf("validate"))]),
                Map([("type", Leaf("field")), ("name", Leaf("db"))]),
            ]))
        ])
    ]))
])
```

Queryable with the constraint walker:
```sprf
rule(service_methods) {
  sem({ entities: [...{ type: "class", name: "$NAME_Service", children: [...{ type: "method", name: $METHOD }] }] })
};
```

## Per-file integration detail

sem's `entities` command works on a single file. During sprefa's per-file scan:

1. For each file, call sem's entity extraction (library call, not CLI)
2. Get back `Vec<SemanticEntity>` with types, names, hashes, parent_ids
3. Flush as rows in a `sem_entity` per-rule table
4. After all files scanned, build EntityGraph from the entity table (pass 2: token-based reference resolution)
5. Flush edges to an `entity_dep` link table
6. Blast radius queries are now SQL over those tables

The normalization opportunity: sem and sprefa both extract from the same files. sem gives entity-level identity (function "foo"), sprefa gives string-level refs (the string "foo" at byte 1234). Joining them:

```sql
-- "this string ref is inside this semantic entity"
SELECT e.entity_type, e.name, r.value, r.byte_start
FROM sem_entity e
JOIN refs r ON r.file_id = e.file_id
  AND r.byte_start >= e.start_byte
  AND r.byte_end <= e.end_byte
```

This answers: "the import of lodash at line 3 is inside the function validateUser." Blast radius + import tracking in one query.

## Recommended approach

Path 2 (per-file extractor) for entity extraction. Path 4 (TreeNode adapter) for querying. This gives:

- Entity-level granularity during normal scans
- Blast radius queries via SQL
- Cross-repo entity linking (sprefa's core differentiator)
- Constraint language patterns over entity trees
- No CLI subprocess overhead
- Shared tree-sitter infrastructure

The `cmd()` path (Path 1) is the zero-effort starting point if we want to experiment before committing to a crate dependency.
