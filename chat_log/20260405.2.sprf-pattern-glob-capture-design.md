# Session: sprf-pattern-glob-capture-design

## Goal
Evolve the sprf language to support advanced pattern matching with glob captures that can be converted to regex, enabling scoped/nested rule extraction.

## Current State
- sprf already supports nested block syntax: `repo($REPO) { rev(main) { folder(packages/$PKG) { ... } } }`
- Parser in `_1_parse.rs` handles RuleBody::Block { slot, children } recursively
- AST in `_0_ast.rs` has Slot, Tag, and RuleBody enums
- Current patterns use basic `$VAR` capture in glob strings

## Problem/Context
Need richer pattern matching for paths:
1. Capture multiple segments: `glob/$MATCH/someFolder$$$ALL`
2. Convert glob patterns to regex for `re::` patterning
3. Support "cut" semantics - where patterns stop matching
4. Make Pattern abstraction ambidextrous (works for both glob and regex)

## Solution/Approach
Extend the Pattern type to handle:
- Single `$VAR` capture in paths (already works)
- Multi-segment capture: `$$$ALL` captures rest of path
- Named capture groups that work with both glob and regex
- Integration with existing `re::` patterns in sprf

## Key Insights
1. The nested block syntax already works in parser - no need to reinvent
2. Pattern abstraction should unify glob and regex (ambidextrous)
3. Scoped blocks create variable bindings visible to children via scope arena
4. Glob captures like `$VAR` need to become regex capture groups `(?P<VAR>...)`

## Open Questions

- How to handle path separators in captures?
- Should we support alternation in globs like `main|release/*`?
