## Architecture Decisions (this session)

### Crate split: index vs cache
- `crates/index` -- pure Rust, no sqlx, no async. git2 tree walk + blob reading, rayon extract, in-memory resolver. Output: `Vec<FileRefs>`
- `crates/cache` -- SQLite progressive cache. Takes `Vec<FileRefs>`, writes deltas.
- Testing extraction requires no DB setup at all.

### File enumeration: libgit2
- Replace `git ls-files` subprocess + mmap with `git2::Repository::open()` + tree walk + `blob.content()`
- Works on bare clones AND normal clones transparently
- Sister git tool uses bare clones in staging dir

### Git tool integration
- Sister tool (TOML config) manages repo syncing, PR tracking, branch head polling, bare clones
- Sprefa reads git tool TOML config for repo paths/names
- Sprefa reads git tool SQLite for changed files / branch HEAD hashes
- No `sprefa add` needed for managed repos
- Integration: git tool detects branch update → calls `sprefa scan --repo X --branch Y`

### Pre-scan context
- `pre_scan(db, repo_ids) -> ScanContext { known_files, known_strings, repo_ids }`
- Load per-repo (not globally) to bound memory
- Extract phase checks `known_files` to skip unchanged (zero DB)
- Flush phase writes only deltas

### Path refs and target_file_id
- `is_path = true` refs: identity is `target_file_id`, not `strings.value`
- `"../utils"` from two files is NOT a shared string -- they point to different targets
- Resolver runs post-scan in Rust, fills `target_file_id` before flush
- Path refs that resolve successfully don't need strings rows
- Query "who imports file X" uses `WHERE target_file_id = ?`, not string join

### Match table (planned, not yet implemented)
- Match is first-class: `matches (id, rule_name, file_id, node_path, scan_run_id)`
- Refs get `match_id` (nullable) + `capture_name` (nullable)
- JS extractor refs: `match_id = NULL`, eventually `rule_name = "oxc_internal::import_declaration"`
- `parent_key_string_id` becomes redundant once match table exists
- Rules table: `rules (id, name, fingerprint, source_text, archived_at)` for rule versioning

## URTSL Design (this session)

### Selector syntax
- Path/glob layer (branch > repo > file): CSS/glob rhythm -- `*`, `**`, `{a,b}`, `*.yaml`
- Node layer (below file segment): ast-grep metavariable convention
  - `$VAR` = one segment, named
  - `$$$VAR` = any depth, named
  - `$_` = one segment, unnamed wildcard
  - `$$$` = any depth, unnamed
- Glob is syntactic sugar that desugars to regex before compilation
- `**` and `$$$` are walk multiplicity operators, not patterns -- engine-driven, not regex

### @ast-grep escape hatch
- `@ast-grep` block with YAML block scalar (`|`) for pattern
- Captures referenced by metavar name in properties block
- For JS/TS: uses oxc not tree-sitter -- block becomes `@oxc` eventually
- oxc AST is just enums, patterns are specific variant arms, codegen is tractable

### Build-time codegen
- `build.rs` reads YAML rules, generates Rust via `include!(concat!(env!("OUT_DIR"), "/rules.rs"))`
- File selector: glob → regex, static match arm
- `@ast-grep` pattern: embed as `&str` constant, wire capture bindings
- `filter:` on capture: embed regex as `&str` constant
- Fully compiled version generates actual oxc visitor arms
