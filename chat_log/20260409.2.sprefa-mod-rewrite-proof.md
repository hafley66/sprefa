# Session: sprefa-mod-rewrite-proof

## Goal
Close the watcher's mod declaration rewrite gap, fix branches/revs mismatch in `add` command, and prove end-to-end with a live watcher test.

## Current State
All changes implemented, tested (415 tests pass), and proven live. `mv src/types.rs src/0_types.rs` now rewrites `mod types;` → `mod 0_types;` in lib.rs, plus all `use crate::types::*` paths in both lib.rs and utils.rs.

## Problem/Context
1. Watcher detected Rust file moves but only rewrote `use` paths, leaving `mod` declarations stale.
2. `sprefa add` wrote `branches = ["main"]` but config struct expects `revs` -- silently ignored by serde.
3. During live proof, discovered two pre-existing bugs: (a) hash algorithm mismatch between scanner (git blob OID) and watcher (xxh3) prevented move correlation, (b) `create_rule_tables` with `None` hashes DROPped builtin tables on every watcher startup, wiping scan data.

## Solution/Approach
- Option A (new planner phase): added mod declaration rewrite step inside `plan_file_move`, between intra-crate RS_USE rewrites and cross-crate rewrites
- Strategy 1 (rewrite all refs): rewrite all internal `use crate::` paths rather than generating re-export shims
- Hash fix: changed watcher's `hash_file` to compute git blob SHA-1 (`sha1` crate added)
- Table wipe fix: split `need_drop` into `need_drop` (schema changed) and `need_ensure` (no hash info), latter uses CREATE IF NOT EXISTS

## Key Insights
- Scanner stores git blob OIDs as content_hash (from git2 tree walk), watcher must match this for move correlation.
- `create_rule_tables` with `None` hashes treated builtins as "schema changed" → DROP + CREATE. Safe during scan (data re-inserted), destructive during watch/daemon startup.
- `mod_parent_candidates` must return repo-relative paths (e.g., `src/lib.rs`) to match DB `files.path` storage, not absolute paths.
- macOS fsevents can report 6 events for a single `mv` (create + 2 removes + metadata). The debounce + classify pipeline handles this correctly.
- The `_ => {}` catch-all in `plan_decl_rename` for RS_MOD still exists. The reverse direction (user edits `mod types;` directly → rename file) is a separate feature.

## Open Questions
- Should the decl-rename direction for RS_MOD (user renames `mod types;` → `mod 0_types;`) also rename the file on disk? Bigger scope.
- The `add` command's generated revs always defaults to `["main"]`. Should it detect the actual default branch?
