## Key Insights
- Per-rule tables store repo+rev in the same row, eliminating the cartesian product that plagued the old file_id-based match_labels discovery join
- `repo()`/`rev()` serve dual purpose: scoping tags (context filter) AND inline annotations (scan target markers inside json patterns)
- The `ScanAnnotation` threading approach (mutable vec through parser functions) avoids changing `SelectStep` types which are serde-serialized
- `unscanned_rev_pairs` uses NOT EXISTS subquery against repo_revs to find pairs not yet scanned
- Old `flush()` in flush.rs is dead code but `delete_rev_files_by_paths` and `rename_file_paths` from same file are still used by SqliteStore
