## Problem/Context
User wants sprefa to eventually represent complex cross-repo causal chains: "here are all the rules that say this openapi spec from this repo is meant to be found and equal the same string as this version string over in this file that gets copied from the source of truth repo and must match" -- and derive consumer patterns like useOPNAMEQuery from RTKQ. This is the motivating use case for the link DSL and cross-repo scope work.

User also described the data flow: app packages reference library versions via package-lock.json, library versions come from git tags, and the reference path goes through URL matching on repo names. This implies:
1. Link rules need to chain: tag -> version string -> dep reference -> consumer pattern
2. The DSL needs to be expressive enough to capture these multi-hop chains
3. Cross-repo scoping matters because the source-of-truth repo and consumer repo are different

## Solution/Approach

### Commit 4: Link rule DSL (next up)
Replace raw SQL WHERE injection with a bounded predicate language. Keep sql as escape hatch.

Types to add in crates/rules/src/types.rs:
```rust
#[serde(tag = "op", rename_all = "snake_case")]
pub enum LinkPredicate {
    KindEq { side: Side, value: String },  // src_m.kind = 'X' or tgt_m.kind = 'X'
    NormEq,                                 // src_s.norm = tgt_s.norm
    Norm2Eq,                                // src_s.norm2 = tgt_s.norm2
    TargetFileEq,                           // src_r.target_file_id = tgt_r.file_id
    StringEq,                               // src_r.string_id = tgt_r.string_id
    SameRepo,                               // src_f.repo_id = tgt_f.repo_id
    And { all: Vec<LinkPredicate> },        // conjunction
}
pub enum Side { Src, Tgt }
```

Update LinkRule: sql becomes Option<String>, add predicate: Option<LinkPredicate>. Exactly one must be set.

New file crates/rules/src/link_compile.rs: pub fn compile(pred: &LinkPredicate) -> String. Recursive match producing SQL fragments.

Update resolve_match_links in crates/cache/src/match_links.rs: if rule.predicate is Some, compile it; else use rule.sql.

Migrate sprefa-rules.json rules to predicate syntax:
- import_binding: And[KindEq(src, import_name), KindEq(tgt, export_name), TargetFileEq, StringEq]
- image_source: And[KindEq(src, image_repo), KindEq(tgt, package_name), NormEq]

### Commit 5: Cross-repo linking scope
Add target_repos: Option<Vec<String>> to LinkRule. When set, adds JOIN repos tgt_rp ON tgt_f.repo_id = tgt_rp.id and AND tgt_rp.name IN (...) to the match_links skeleton. No change when None.

### Commit 6: ghcache integration improvements

## Key Insights
- SQLite does NOT support data-modifying CTEs (DELETE inside WITH). Temp tables are the workaround for cascade deletes.
- User is extremely strict about no SQL in loops, no N+1 queries, no exceptions. All batch operations must use temp tables or single statements.
- ExtractContext.tags is &[&str] (borrowed), so tags must be read before the spawn_blocking that does extraction, or inside the same blocking task. Current approach: read in a separate spawn_blocking before extraction.
- The git_tags table already existed in schema (migrations.rs:110-120) with GitTag type (types.rs:183-192) but had no query functions and was never populated. Now wired through the full pipeline.
- The deploy-image rule reuses the Object select step already tested in extractor.rs:285-296. Just needed different kind strings.
- sprefa_schema is only a dev-dep of the scan crate. Tag persistence goes through sprefa_cache::flush_git_tags wrapper (same pattern as SHA persistence going through the CLI caller).
- The match_links SQL skeleton scopes source to one repo but leaves target unconstrained. This is an intentional design for cross-repo linking, but commit 5 adds opt-in target_repos scoping.
- User's long-term vision: multi-hop causal chains (openapi spec -> version string -> dep reference -> RTKQ consumer pattern like useOPNAMEQuery). The link DSL and cross-repo scope are building blocks for this.

## Open Questions
- Link DSL: should there be an Or combinator in addition to And? Current plan only has And.
- User mentioned "random url matching on some repo name" for version references -- may need a Regex or Contains predicate variant in the DSL beyond the initial set.
- Multi-hop link chains: current match_links is single-hop (source_match -> target_match). Multi-hop would require either recursive resolution or a separate chain table. Not in scope for this session but worth noting.
- RTKQ pattern derivation (useOPNAMEQuery from operationId): this is a string transform rule (PascalCase + prefix/suffix). The current system matches exact/normalized strings. Pattern-based transforms would be a new capability.
